## Detailed Risk Register

### TECH-001: Formula Delimiter Counting Edge Cases

**Category:** Technical
**Probability:** Low (1) - Edge cases are rare
**Impact:** Medium (2) - Could cause incorrect metadata
**Score:** 2 (Low Risk)

**Description:**
The formula counting algorithm uses string.count('$$') and string.count('$') which could theoretically be confused by escaped delimiters or malformed LaTeX.

**Affected Components:**
- `controller.py::_embed_formulas_in_chunks()`

**Mitigation Strategy:**
- **Type:** Preventive + Detective
- **Actions:**
  - Current algorithm handles standard LaTeX well (tested)
  - Future: Add validation for balanced delimiters if needed
  - Test coverage includes multiple formulas per chunk
- **Testing:**
  - Unit test: test_embed_formulas_multiple_in_chunk
  - Edge case: test_inline_vs_display_delimiters
- **Residual Risk:** Very Low - Standard LaTeX usage is well-covered

**Status:** Mitigated

---

### TECH-002: Reading Order Heuristic for Formula Placement

**Category:** Technical
**Probability:** Medium (2) - May occur with complex layouts
**Impact:** Medium (2) - Formulas might be slightly mispositioned
**Score:** 4 (Medium Risk)

**Description:**
Current MVP implementation appends formulas at end of document by page rather than inserting at precise text positions. This is intentional for MVP but could cause reading order issues in complex documents.

**Affected Components:**
- `controller.py::_merge_text_and_formulas()`

**Mitigation Strategy:**
- **Type:** Accepted for MVP + Future Enhancement
- **Actions:**
  - MVP: Group formulas by page and append (simple, reliable)
  - Comment in code notes this is MVP approach
  - Future: Parse text structure and insert at bbox positions
- **Testing:**
  - Test: test_merge_text_and_formulas_reading_order
  - Validates page-level ordering works correctly
- **Residual Risk:** Medium - Complex layouts may have sub-optimal formula placement
- **Timeline:** MVP accepted; enhance in v0.3 if needed

**Status:** Accepted (documented limitation)

---

### PERF-001: String Counting Performance for Large Documents

**Category:** Performance
**Probability:** Low (1) - Most documents are moderate size
**Impact:** Low (1) - Slight performance degradation
**Score:** 1 (Minimal Risk)

**Description:**
The `_embed_formulas_in_chunks()` function uses string.count() which is O(n) per chunk. For very large documents with many chunks, this could add processing time.

**Affected Components:**
- `controller.py::_embed_formulas_in_chunks()`

**Mitigation Strategy:**
- **Type:** Acceptable + Monitor
- **Actions:**
  - O(n) is acceptable for MVP (string.count is highly optimized in Python)
  - Performance tests show no issues with typical documents
  - Future: Profile and optimize if needed (compiled regex, etc.)
- **Testing:**
  - Performance validated in test suite (4 seconds for 41 tests)
  - No performance degradation observed
- **Residual Risk:** Very Low - Unlikely to be a problem in practice

**Status:** Accepted (monitor in production)
