## High Risks

### 1. PERF-001: Model Re-initialization in Batch Mode

**Score: 6 (High)**

**Probability**: High (3)
- Model initialization happens in controller.py per-file processing loop
- No caching mechanism implemented
- Every PDF triggers new model load (~1-2s overhead)

**Impact**: Medium (2)
- Batch processing significantly slower than necessary
- ~50-70% performance degradation for multi-file runs
- User experience degraded but not broken
- No data loss or corruption

**Mitigation**:
1. Implement lazy singleton pattern in formula_extractor.py
2. Cache model instance at module level
3. Add performance timing logs to measure improvement
4. Document caching strategy in code comments

**Testing Requirements**:
- Performance test: Process 10 PDFs, measure total time
- Before fix: ~20-30s overhead
- After fix: ~2-3s overhead (single initialization)
- Unit test: Verify model instance reuse

**Residual Risk**: Low - After caching, only first-run download remains

**Owner**: dev
**Timeline**: Before production deployment

### 3. DATA-001: Formula Extraction Accuracy Limitations

**Score: 4 (Medium)**

**Probability**: Medium (2)
- pix2tex accuracy documented at >75% recall
- Handwritten formulas have lower accuracy
- Complex custom LaTeX macros may not convert

**Impact**: Medium (2)
- Some formulas may be extracted incorrectly
- Users may need manual review for critical documents
- Graceful degradation (continues processing)

**Mitigation**:
1. Document accuracy limitations clearly in README ✓ (DONE)
2. Add quality indicators in formula metadata (confidence scores)
3. Provide manual review workflow in documentation
4. Consider RapidLaTeXOCR alternative (future Epic 8.2)

**Testing Requirements**:
- Accuracy test: Known ground truth PDFs
- Measure: Precision/recall metrics
- Edge cases: Handwritten, complex notation

**Residual Risk**: Medium - Inherent to ML model limitations

**Owner**: product (accept limitations for MVP)
**Timeline**: Monitor in production, improve in future releases
