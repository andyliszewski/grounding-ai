### RISK-002: External Dependencies (Java + Audiveris)
**Severity:** LOW
**Likelihood:** MEDIUM
**Impact:** LOW
**Category:** Technical - Dependencies
**Status:** MITIGATED

**Description:**
The OMR parser requires two external dependencies not managed by pip:
1. Java Runtime Environment (JRE) >=11
2. Audiveris >=5.3

**Evidence:**
- JRE check: omr_parser.py:96-123 (`_check_java_version()`)
- Audiveris check: omr_parser.py:56-94 (`_find_audiveris_binary()`)
- Installation guide referenced in error messages

**Impact Analysis:**
- Users must manually install JRE and Audiveris
- Installation complexity increases
- Platform-specific installation paths (macOS, Linux, Windows)

**Mitigation Implemented:**
1. **Graceful error handling:**
   - Clear error messages with installation instructions
   - Links to download sites (adoptium.net for JRE, GitHub for Audiveris)
   - Error messages reference installation guide (docs/epics/epic-7-installation-guide.md)

2. **Flexible detection:**
   - Checks AUDIVERIS_HOME environment variable
   - Checks macOS default path
   - Checks PATH for audiveris binary

3. **Tests don't require dependencies:**
   - All tests use mocking (subprocess, music21)
   - CI/CD can run tests without installing Java/Audiveris

**Evidence of Mitigation:**
```python
# omr_parser.py:160-163
raise AudiverisOMRError(
    "Java Runtime Environment (JRE) >= 11 is required for Audiveris. "
    "Install from https://adoptium.net/ or use 'brew install openjdk' on macOS."
)
```

**Recommendation:** ACCEPT - Appropriate error handling and documentation provided. Risk is inherent to using Audiveris (unavoidable).

### RISK-003: MusicXML Parsing Errors
**Severity:** LOW
**Likelihood:** LOW
**Impact:** MEDIUM
**Category:** Technical
**Status:** MITIGATED

**Description:**
Audiveris may generate malformed or incomplete MusicXML output for complex scores, which could cause music21 parsing failures.

**Evidence:**
- music21 parsing in omr_parser.py:236-288
- Error handling wraps entire parsing block: lines 249-287

**Impact Analysis:**
- Complex scores (multiple instruments, non-standard notation) may fail
- music21 exceptions could crash processing
- Invalid MusicElement objects could propagate downstream

**Mitigation Implemented:**
1. **Try-catch wrapper:**
   ```python
   try:
       score = converter.parse(str(musicxml_path))
       # ... processing ...
   except Exception as e:
       raise AudiverisOMRError(f"Failed to parse MusicXML with music21: {e}")
   ```

2. **Input validation:**
   - File existence check (line 149-150)
   - PDF format validation (line 155-156)
   - MusicXML output existence check (line 220-224)

3. **Dataclass validation:**
   - MusicElement uses Literal types for element_type (line 38)
   - Optional types for pitch/duration prevent None errors

4. **Tests cover edge cases:**
   - test_parse_music_pdf_no_musicxml_output (line 277)
   - test_parse_music_pdf_audiveris_fails (line 249)

**Recommendation:** ACCEPT - Comprehensive error handling implemented. Story 7.1 PoC validated MusicXML quality (104 notes extracted successfully).
