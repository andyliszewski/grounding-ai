## Story

**As a** grounding-ai contributor,
**I want** a pure, testable reranker module that wraps a sentence-transformers cross-encoder with singleton caching and a generic re-ordering API,
**so that** downstream stories (18.2/18.3/18.4) can wire it into `SearchCorpusTool`, the MCP server, `local_rag.py`, and the eval harness without further design decisions.

---

## Acceptance Criteria

1. A new module `grounding/reranker.py` exports `RerankConfig` (dataclass) and `CrossEncoderReranker` (class) plus a module-level `rerank(query, chunks, *, config, text_key)` convenience function.
2. `RerankConfig` is a frozen dataclass with fields: `enabled: bool = False`, `model: str = "BAAI/bge-reranker-base"`, `pool_size: int = 50`, `batch_size: int = 16`. `validate()` method rejects `pool_size < 1`, `batch_size < 1`, and empty `model`.
3. `CrossEncoderReranker.rerank(query: str, chunks: Sequence[Mapping[str, Any]], *, text_key: str = "content") -> list[dict]` returns a new list of dicts re-ordered by cross-encoder score, preserving every key from the input dicts.
4. The returned dicts gain two new fields and mutate one:
   - New: `rerank_score: float` â€” cross-encoder raw score.
   - New: `faiss_distance: float` â€” the original `score` value from the input (preserved).
   - Mutated: `score` â€” replaced with `rerank_score` so downstream consumers see the cross-encoder score under the existing key.
5. Empty input short-circuits: `rerank(query, [])` returns `[]` without loading the model.
6. Model loading is lazy and cached: a process-level singleton keyed on `RerankConfig.model` ensures the model loads at most once per unique model name per process.
7. Loading follows the retry pattern already used in `grounding/embedder.py` (up to 3 attempts, clear error log, `RuntimeError` on final failure).
8. CPU-only: no CUDA assumptions. Model runs on CPU regardless of CUDA availability.
9. Determinism: identical `(query, chunks, config)` inputs produce identical output ordering and scores across invocations within a process.
10. Unit tests in `tests/test_reranker.py` cover: empty input, single-chunk no-op ordering, multi-chunk reordering via a stub CrossEncoder, determinism across repeated calls, config validation, singleton caching (model factory called once), text_key override. No network access, no real model download.
11. No new runtime dependencies. `sentence-transformers` is already in the project; `CrossEncoder` is reachable via `from sentence_transformers import CrossEncoder`.
12. Module-level `rerank(query, chunks, *, config, text_key="content")` convenience function instantiates (or reuses) a singleton `CrossEncoderReranker` from the given config and calls `.rerank(...)` on it.

### Dict-in / Dict-out design rationale

The retrieval path works in raw dicts. The eval path uses `RetrievedChunk` dataclasses. Making the reranker generic over dicts (with a configurable `text_key`) keeps it decoupled from either. When 18.2 wires it into retrieval, no conversion. When 18.4 wires it into eval, a small `asdict(chunk)` / `RetrievedChunk(**d)` bridge lands there, not here.

### Singleton cache shape

```python
_model_cache: dict[str, CrossEncoder] = {}
```

Keyed on model name (not full config) because the model is the only expensive load. `pool_size`, `batch_size`, `enabled` are per-call choices that don't require a separate model instance.

### Why replace `score` with `rerank_score` (rather than adding alongside)

Downstream consumers (agent prompts, eval reports, MCP clients) read `score` as "how relevant is this chunk to the query." After reranking, the cross-encoder score *is* the best answer to that question. Preserving the FAISS distance under a new key lets debugging and eval reports show both, without asking every consumer to learn a new field name.

### CPU-only invariant

Force `device="cpu"` at model load. Rationale: determinism (no per-host variation depending on CUDA state), and the latency budget in Epic 18's dev notes assumes CPU. If GPU becomes interesting later, it's a Tier-3 follow-up with its own feature flag.
