## Story

**As a** grounding-ai user,
**I want** every chunk written to disk to carry correct `page_start`, `page_end`, and `section_heading` in its YAML front matter,
**so that** downstream retrieval (17.3) and eval (17.4) have real citation data to surface, and so the on-disk corpus becomes immediately more useful even before retrieval is rewired.

---

## Acceptance Criteria

1. A new function `split_markdown_with_map(text, elements, config) -> list[ChunkWithProvenance]` lives in `grounding/chunker.py`. Each `ChunkWithProvenance` carries `text: str`, `char_start: int`, `char_end: int`.
2. The existing `split_markdown(text, config) -> List[str]` is preserved bit-for-bit for callers that do not need provenance.
3. A new helper `derive_chunk_metadata(chunk, elements) -> ChunkProvenance` lives alongside and returns `page_start: int | None`, `page_end: int | None`, `section_heading: str | None`.
4. Determinism: the same `(text, elements, config)` input produces identical `ChunkWithProvenance` outputs across runs and across machines.
5. A chunk whose char range overlaps elements on pages 3â€“5 gets `page_start=3, page_end=5`. Elements with `page_number=None` are excluded from the min/max computation. A chunk where every overlapping element has `page_number=None` gets `page_start=None, page_end=None`.
6. `section_heading` is the last element of the `heading_stack` of the element at or before `chunk.char_start`. When the chunk starts before any heading, `section_heading=None`. When a chunk straddles a heading boundary, the **earlier** section wins; this convention is documented in the function docstring and in `CLAUDE.md`.
7. `grounding/controller.py:533` passes the derived `page_start`, `page_end`, `section_heading` into `build_chunk_metadata`. On-disk chunk front matter is populated for PDF input; null-valued but present for Markdown/EPUB where fields don't apply.
8. The controller uses `format_markdown_with_map` from Story 17.1 to obtain elements for PDFs. If the parser path is `TextElement` (pdftotext fallback), the element map has `page_number=None` everywhere and all chunks land with null pages â€” no crash, no warning spam.
9. Unit tests in `tests/test_chunker_provenance.py` cover: single-page chunk, chunk spanning pages 3â€“5, chunk spanning a heading boundary (earlier section wins), chunk before any heading (null section), Markdown-only input (null pages, populated sections), pdftotext fallback (null everywhere). Six tests minimum.
10. Integration test runs the real pipeline against a small committed test PDF (1-3 pages) with known page/section structure and asserts chunk front matter on disk matches hand-verified expectations.
11. Epic 16 mini-corpus eval gate stays green on this story's PR (no retrieval regression).

### Char-offset derivation

LangChain's `RecursiveCharacterTextSplitter.split_text` returns chunks but not offsets. Options:
1. **Running cursor scan (preferred):** After split, iterate chunks in order. For each chunk `C`, find the first occurrence of `C` in `text` starting at `cursor`. Set `char_start = cursor_match_position`, `char_end = char_start + len(C)`, `cursor = char_end - overlap`. Deterministic.
2. Fork LangChain's implementation. Not worth it.

Edge case: duplicate paragraphs in the source (two chunks might be byte-identical). The cursor scan handles this correctly because it always starts the search from the current cursor, not from 0.

### Boundary convention, restated

"Earlier section wins" applied to a chunk starting mid-Â§3.2 and ending mid-Â§3.3 means `section_heading = "3.2 ..."`. Rationale: a reader arriving at the start of the chunk is still inside Â§3.2; that's the context in which the passage should be cited. If the passage contains content truly belonging to Â§3.3, the chunker's `chunk_size` / `chunk_overlap` settings are likely too coarse â€” a separate concern.

### Pages as ranges vs singletons

If `page_start == page_end`, keep both fields (don't collapse to a single `page`). This keeps the schema regular and lets the 17.3 formatter decide whether to render `p.247` vs `p.247â€“249`.

### Null behavior is a feature

Chunks from Markdown-only docs get `page_start=null, page_end=null, section_heading="..."`. Chunks from scanned-PDF pdftotext fallback get all three null. Both are valid on-disk states; the 17.3 citation formatter and the 17.4 eval harness both handle nulls.

### Performance Considerations

`derive_chunk_metadata` is O(n) over elements per chunk, giving O(nÂ·m) for a document with `n` elements and `m` chunks. For typical corpus documents (hundreds of elements, tens of chunks), this is negligible. The story's Dev Notes flag binary search as an alternative; linear scan is appropriate here.

The section-heading loop continues scanning past `chunk.char_end` until an element's `char_start` exceeds `chunk.char_start` (which typically happens early). In practice the loop terminates quickly because most chunks start near an element boundary. Fine as-is.
