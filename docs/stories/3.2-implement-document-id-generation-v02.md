## Story Description

As a **developer**, I want to **derive stable document identifiers from formatted Markdown artifacts** so that downstream chunk metadata, manifests, and deduplication workflows can reference documents deterministically.

This story introduces a doc ID generator that leverages SHA-1 hashes of the canonical Markdown output (from Story 2.3) and integrates the identifier with pipeline context to make it accessible for later metadata and manifest steps.

---

## Context

### From PRD v0.2  
[Source: docs/prd.md — Sections "Metadata Requirements" & "Observability"]

- Each document requires a short, stable identifier for manifest entries and chunk metadata.  
- IDs must be reproducible across runs when the underlying content is unchanged.  
- Collisions must be observable and logged for manual remediation.

### From Architecture v0.2  
[Source: docs/architecture.md — Sections 5 File Context & 7 Chunking]

- File context should carry `doc_id`, `slug`, and hashes for downstream consumers.  
- Doc IDs feed into chunk YAML front matter and manifest entries.  
- Hashing utilities should live in a shared module (e.g., `pdf2llm/stats` or new `hashing.py`) for reuse.

### From Epic 3  
[Source: docs/epics/epic-3-chunking-metadata-v02.md — Story 3.2]

- Generate doc ID from SHA-1 hash (first 8 hex characters).  
- Integrate slug + hash into pipeline context.  
- Warn on potential collisions.

---

## Acceptance Criteria

### AC1: Hashing Utility
- ✅ Implement `compute_sha1(data: bytes | str) -> str` returning full 40-character hex digest.  
- ✅ Provide helper `short_doc_id(sha1: str) -> str` yielding first 8 hex characters.  
- ✅ Inputs support both strings (UTF-8) and bytes.

### AC2: Pipeline Integration
- ✅ Pipeline attaches `doc_id` to `FileContext` alongside existing slug/sha.  
- ✅ Doc ID derived from formatted Markdown (post-normalization) to ensure deterministic output.  
- ✅ `doc_id` exposed to subsequent metadata/chunking steps.

### AC3: Collision Detection & Logging
- ✅ If two documents produce identical doc IDs within a run, log a warning with source filenames and full SHA-1.  
- ✅ Provide hook (e.g., set of seen doc IDs) to detect collisions during pipeline execution.  
- ✅ No pipeline abort; collision information returned in stats or result bundle for observability.

---

## Implementation Guidance

1. **Utility Module:** Add `pdf2llm/hashing.py` (or extend `stats.py`) with reusable hashing helpers; unit-test in isolation.  
2. **Pipeline Update:** After formatting (Story 2.4), compute SHA-1 of normalized Markdown, store full hash + short ID on `FileContext`.  
3. **Collision Tracking:** Maintain a set/dict of `doc_id -> sha1` during pipeline run; on collision, log `pipeline` warning and append note to stats.  
4. **Exposure:** Ensure pipeline return object includes doc IDs so chunker/metadata stories can consume them without recomputing.  
5. **Documentation:** Update README/architecture notes if new module introduced; note doc ID generation in story docs.
