# Story 6.6: MCP Corpus Search Server

**Epic:** Epic 6 - Vector Embeddings & Semantic Search
**Status:** Done
**Owner:** Scrum Master (Bob)
**Created:** 2025-02-22
**Dependencies:** Story 6.3 (Vector Store), Story 6.4 (Query Interface)

---

## Story

**As a** corpus-powered agent,
**I want** semantic search exposed as an MCP tool,
**so that** I can query my filtered knowledge base directly from conversations without manual CLI invocation.

---

## Background

Epic 6 delivered FAISS vector indexes and a CLI query interface, but agents couldn't use them natively. The gap identified:

- FAISS indexes generated for each agent (embeddings/<agent>/_embeddings.faiss)
- Query interface existed (`python -m pdf2llm.query`)
- **Missing**: No programmatic integration point for Claude Code agents

This story closes that gap by exposing corpus search as an MCP (Model Context Protocol) server, enabling agents to search their filtered corpora as a native tool call.

---

## Acceptance Criteria

1. MCP server starts without errors when configured in .mcp.json
2. `search_corpus` tool returns relevant chunks from agent's filtered FAISS index
3. Results include source attribution (doc_id, chunk_id, source filename)
4. `list_corpus_agents` tool lists all agents with available embeddings
5. Index caching prevents redundant disk reads across queries
6. Graceful error handling for missing agents or indexes
7. Similarity scores normalized to 0-1 range (higher = more relevant)
8. All corpus agent slash commands include instructions to use `search_corpus` tool

## Dev Notes

### Architecture

The MCP server integrates with the existing vector embeddings infrastructure:

```
Agent Request
    ↓
search_corpus(query, agent, top_k)
    ↓
┌─────────────────────────────────────┐
│  MCP Server (mcp_servers/corpus_search/server.py)
│  ├── Lazy-load SentenceTransformer
│  ├── Load/cache FAISS index for agent
│  ├── Embed query → 384-dim vector
│  ├── FAISS search → indices + distances
│  ├── Read chunks from corpus/<slug>/chunks/
│  └── Format as markdown with sources
└─────────────────────────────────────┘
    ↓
Markdown results with source attribution
```

### File Structure

```
mcp_servers/
├── __init__.py                    # Package marker
└── corpus_search/
    ├── __init__.py                # Subpackage marker
    ├── server.py                  # MCP server implementation (340 lines)
    └── requirements.txt           # Dependencies: mcp, sentence-transformers, faiss-cpu

.mcp.json                          # Claude Code MCP configuration
```

### Key Implementation Details

**Embedding Model Reuse** (server.py:56-63):
- Uses same model as ingestion: all-MiniLM-L6-v2
- Lazy-loaded on first query to avoid startup delay
- Global singleton pattern for efficiency

**Index Caching** (server.py:66-99):
- Agent indexes cached in `_index_cache` dict
- Handles both legacy list format and new dict format with "chunks" key
- Cache persists for lifetime of server process

**Similarity Score Conversion** (server.py:174-176):
- FAISS returns L2 distances (lower = more similar)
- Converted to 0-1 similarity via `exp(-distance)`
- Aligns with user expectation (higher = better)

**Result Formatting** (server.py:206-227):
- Markdown format for LLM context injection
- Includes rank, score, source filename
- Preserves doc_id and chunk_id for citation

### Configuration (.mcp.json)

```json
{
  "mcpServers": {
    "corpus-search": {
      "command": "/path/to/venv/bin/python",
      "args": ["-m", "mcp_servers.corpus_search.server"],
      "cwd": "/path/to/grounding-ai",
      "env": {
        "CORPUS_DIR": "/path/to/Corpora/corpus",
        "EMBEDDINGS_DIR": "/path/to/Corpora/embeddings",
        "AGENTS_DIR": "/path/to/grounding-ai/agents"
      }
    }
  }
}
```

### Test Cases

1. **Search with valid agent**: Returns ranked chunks with scores
2. **Search with invalid agent**: Returns error with list of available agents
3. **Search with empty query**: Returns validation error
4. **List agents**: Returns sorted list of agents with embeddings
5. **Multiple queries**: Index cached, second query faster

### Known Limitations

- No incremental index updates (requires full rebuild)
- CPU-only embedding generation
- Single-threaded query processing
- Index loaded fully into memory (may be large for big corpora)
