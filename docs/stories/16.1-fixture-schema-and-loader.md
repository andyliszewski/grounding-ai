## Story

**As a** grounding-ai contributor,
**I want** a documented, validated YAML format for retrieval eval items and a loader that returns typed objects,
**so that** the rest of the eval harness (runner, metrics, CLI, CI) has a stable contract to build against and contributors can add eval items without reading source code.

---

## Acceptance Criteria

1. A fixture YAML schema is documented in `docs/eval/README.md` with one fully worked example.
2. `grounding/eval/fixtures.py` exposes `load_fixtures(path) -> FixtureSet` returning typed dataclasses.
3. Schema validation produces actionable errors (file path, item id, field, reason) — never bare `KeyError`/`TypeError`.
4. The loader resolves `agent` against `agents/<name>.yaml` and fails clearly if the agent is unknown.
5. A schema-example fixture is committed at `docs/eval/fixtures/example.yaml` targeting the bundled `agents/examples/scientist.yaml` agent. The example contains 2-3 items demonstrating the schema (required fields, optional `chunk_ids`, `tags`, `notes`). It uses placeholder `doc_ids` and is **not** intended to score against a real corpus — it exists solely to illustrate the format. The README must label it as such.
6. Unit tests cover: valid file, missing required field, unknown agent, duplicate item id, malformed YAML, empty file.
7. No new runtime dependencies beyond `pyyaml` (already in project).

## Dev Notes

### Fixture Format (canonical example)

```yaml
agent: data-scientist
version: 1
items:
  - id: ds-001
    query: "What does the paper conclude about transformer scaling laws below 1B params?"
    expected:
      doc_ids: ["7a9b2c1f"]
      chunk_ids: ["7a9b2c1f/ch_0023"]   # optional, stricter
    tags: ["methodology", "transformers"]
    notes: "Source: Kaplan et al. scaling laws paper, conclusion section."

  - id: ds-002
    query: "Bootstrap confidence interval procedure for a small sample mean"
    expected:
      doc_ids: ["3c4d5e6f", "9b1a2c3d"]   # any-of match
    tags: ["statistics", "case-study"]
    notes: "Two textbooks cover this; either is an acceptable hit."
```

### Validation Rules

| Field | Required | Type | Rule |
|-------|----------|------|------|
| `agent` | yes | str | Must match `agents/<name>.yaml` filename stem. |
| `version` | yes | int | Currently must equal `1`. |
| `items` | yes | list | Must be non-empty. |
| `items[].id` | yes | str | Unique within file; recommend `<agent-prefix>-NNN`. |
| `items[].query` | yes | str | Non-empty after strip. |
| `items[].expected.doc_ids` | yes | list[str] | Non-empty. |
| `items[].expected.chunk_ids` | no | list[str] | Each must match `<doc_id>/ch_NNNN` pattern. |
| `items[].tags` | no | list[str] | Lowercase kebab-case recommended. |
| `items[].notes` | no | str | Free text. |

### Error Handling Contract

```python
class FixtureValidationError(Exception):
    path: Path
    item_id: str | None
    field: str | None
    reason: str
```

Error message format (single line, suitable for CI logs):
```
docs/eval/fixtures/data-scientist.yaml [item=ds-003 field=expected.doc_ids]: must be non-empty
```

### Why Validate Against `agents/<name>.yaml`

A fixture set targets one agent's view of the corpus. Catching the typo (`data-scientest`) at load time prevents downstream errors that look like "0 retrieval results" rather than "wrong agent name."

## QA Results

### Review Date: 2026-04-13

### Reviewed By: Quinn (Test Architect)

### Code Quality Assessment

Clean, focused, pure-data module. Loader is well-scoped: parse, validate, return frozen dataclasses — no side effects, no corpus/FAISS I/O, no new runtime deps. Validation is exhaustive and error messages are uniformly actionable: every failure path carries `path`, optional `item_id`, optional `field`, and a human-readable `reason`, which gives CI logs and IDE surfaces a stable structured contract. The `UnknownAgentError` subclassing `FixtureValidationError` is a nice touch — lets callers catch either the broad or narrow exception and still get the structured attributes.

Test suite (16 cases) exceeds the AC-required 6 and covers the right boundaries: valid path, each required-field omission, unknown agent, duplicate id, empty list, malformed YAML, empty file, chunk-id regex (good/bad), version gate, missing file, isolation-from-corpus, frozen-dataclass mutation, snapshot of canonical output, and a live round-trip of the bundled `docs/eval/fixtures/example.yaml` — which means any drift between the schema, example, and loader will break CI. That last test is the most load-bearing one for Epic 16's stability guarantee and I'm glad it's present.

### Refactoring Performed

None. Module is small, readable, and test-coverage-gated; the marginal risk of refactoring outweighs the cosmetic wins available.

### Improvements Checklist

- [ ] (nit, dev) `from dataclasses import dataclass, field` — `field` is imported but never used as the dataclasses helper; it's shadowed by the `field` keyword-only parameter on `FixtureValidationError.__init__`. Drop the import to silence future lint noise.
- [ ] (nit, dev) `_CHUNK_ID_RE = r"^[A-Za-z0-9_\-]+/ch_\d{4,}$"` accepts 4-or-more digits, but the docs + field reference table both say `ch_NNNN` (4 digits exactly). Either tighten the regex to `\d{4}` or soften the doc to `ch_NNNN+`. Minor — just pick one and make them agree before Story 16.2 starts matching chunk ids against real FAISS output.
- [ ] (future, po) Schema currently version-pins to `1`. When Story 16.2 introduces runner-side fields (e.g., top-k overrides, negative examples), bumping `SCHEMA_VERSION` should come with a migration note in `docs/eval/README.md`.

### Security Review

Not applicable in the conventional sense — this is a pure YAML parser with `yaml.safe_load`, no deserialization of arbitrary objects, no network, no filesystem writes. Only filesystem reads, confined to the passed fixture path and the existence check on `agents/<name>.yaml`. No injection surface.

### Performance Considerations

Not applicable — fixtures are small (tens to low hundreds of items) and loading is O(n) over items with trivial per-item cost. No caching needed.
