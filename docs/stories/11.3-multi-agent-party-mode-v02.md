# Story 11.3: Multi-Agent Party Mode

**Epic:** Epic 11 - Interactive Corpus Agents
**Status:** Done
**Owner:** TBD

## Story

**As a** knowledge worker with cross-domain questions,
**I want** to activate multiple corpus agents simultaneously via `/party` so they collaboratively discuss a topic from their respective domains,
**So that** I get corpus-grounded, multi-perspective analysis with real citations instead of single-agent answers.

---

## Acceptance Criteria

1. `/party` slash command activates a multi-agent discussion mode with a welcome message and agent roster
2. For each user message, 2-4 most relevant agents are automatically selected based on topic-to-expertise matching
3. Each selected agent searches its own corpus via `mcp__corpus-search__search_corpus(agent=<name>)` before responding
4. Agents respond in-character using their persona style with the format: `[icon] **Agent Name**: response`
5. Responses include corpus-grounded citations marked as `[corpus]`, `[web]`, or `[training]`
6. Agents engage in cross-talk: referencing each other by name, building on points, and respectfully disagreeing
7. Users can optionally request specific agents (e.g., "ask the tax advisor and venture attorney about this")
8. Token management uses a compact agent roster (name, icon, 1-line summary) rather than loading all 23 full personas
9. Clean exit protocol that does not collide with Claude Code's `/exit` command
10. Graceful degradation: agents with missing embeddings still participate using training knowledge (noted via `[training]` citation)
11. Existing individual agent slash commands (`/ceo`, `/scientist`, etc.) remain unaffected
12. No new Python code required; implementation is pure prompt engineering in a single slash command file

## Dev Notes

### Inspiration: BMad Method Party Mode

This feature is inspired by BMad Method's "Party Mode" (see [bmad-code-org/BMAD-METHOD](https://github.com/bmad-code-org/BMAD-METHOD)). BMad's implementation is pure persona role-play within a single LLM context window: an agent manifest is compiled at install time, and the LLM is instructed to select 2-3 relevant agent characters per user message.

**Our critical advantage:** BMad agents have no corpus grounding. Our 23 agents each have FAISS embeddings over domain-specific collections, searchable via the MCP `search_corpus` tool. This means our party mode produces corpus-backed, citable responses rather than role-play.

**Key BMad lessons to adopt:**
- Compact roster format (not full personas) to manage token budget
- 2-3 agent selection per message with diversity rotation
- Cross-talk protocol (agents referencing each other by name)
- Clean exit protocol (BMad had Issue #1179 where "exit" collided with IDE commands)
- Session state tracking for agent rotation

**Key BMad pitfalls to avoid:**
- Token bloat: BMad sessions reach 50-100K tokens; our corpus searches add more context, so we must be aggressive about roster compactness and limit `top_k` to 3
- Lost-in-the-middle: Long party sessions push instructions out of active attention; keep the orchestration prompt concise

The `/party` command embeds a compact roster block (~2K tokens) rather than all 23 full persona blocks (~15K+ tokens). Format per agent:

### MCP Corpus Search Tool Signature

```
mcp__corpus-search__search_corpus(
  agent: str,       # Agent name (e.g., "scientist")
  query: str,       # Search query
  top_k: int = 5    # Number of results (use 3 in party mode to limit context)
)
```

Each selected agent calls this tool with its own name as the `agent` parameter. This routes the search to that agent's filtered FAISS index (`embeddings/<agent>/_embeddings.faiss`).

### Response Format Example

```
User: What are the tax implications of an IP licensing deal structured as an acquisition?

[🦈] **Corp Dev (Hostile)**: The acqui-hire framing is a red flag. When IP is the
real asset, buyers inflate "goodwill" to shift tax burden to the seller...
[Source: Koller, Valuation, corpus]

[📊] **Tax Advisor**: The 338(h)(10) election is critical here. If the deal is
structured as an asset purchase rather than stock purchase, the buyer gets a
stepped-up basis but the seller faces ordinary income on the IP portion...
[Source: IRC Section 338, training] [Source: CA FTB Publication 1001, corpus]

[🛡️] **IP Specialist**: Before structuring, classify the IP: is it a patent
(capital asset, LTCG eligible) or trade secret (ordinary income)? The distinction
drives the entire tax treatment. As Tax Advisor noted regarding 338(h)(10), the
asset vs. stock framing also determines whether patent assignments transfer cleanly...
[Source: MPEP Chapter 300, corpus]
```

### Token Budget Considerations

| Component | Estimated Tokens |
|-----------|-----------------|
| Orchestration prompt (roster + instructions) | ~4,000 |
| Per-agent corpus search results (3 chunks x ~300 tokens) | ~900 per agent |
| Per-agent response | ~300-500 per agent |
| Per round (3 agents, search + response) | ~4,200 |
| 5-turn conversation | ~25,000 (excluding orchestration) |
| **Total for 5-turn session** | **~29,000** |

This keeps well under BMad's 50-100K range while adding real corpus grounding.

### Exit Protocol Design

BMad encountered Issue #1179 where the word "exit" in party mode prompts caused Claude Code to execute `/exit` and terminate the session. To avoid this:

- Use `[E] End discussion` as the menu option (never the word "exit" alone)
- Accept trigger phrases: `end party`, `quit party`, `done`
- On exit: summarize key agreements/disagreements from the session
- Return to normal Claude Code mode (no parent workflow to restore)
