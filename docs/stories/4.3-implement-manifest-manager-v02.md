## Story Description

As a **developer**, I want to **maintain the corpus-level `_index.json` manifest** so that downstream workflows can discover all processed documents, along with summary metadata (doc ID, slug, chunk count, pages).

This story introduces a manifest manager that loads existing manifests, appends or updates document entries, and writes atomically. It leverages metadata produced in Stories 3.x and 4.1/4.2 to populate fields without reprocessing content.

---

## Context

### From PRD v0.2  
[Source: docs/prd.md — Sections "Corpus Management" & "Batch Processing"]

- Users expect a single manifest summarizing processed documents for ingestion pipelines.  
- Manifest must persist across runs and be append-only (unless document reprocessed).  
- Corruption must be avoided; atomic writes required.

### From Architecture v0.2  
[Source: docs/architecture.md — Sections 5 Data Contracts & 6 Output Writer]

- `_index.json` schema includes `created_utc`, `docs` list with entries: `doc_id`, `slug`, `orig_name`, `pages`, `strategy`, `chunk_count`.  
- Stored at root of output directory.  
- Entries should be deterministic and sorted (e.g., by slug or doc_id).

### From Epic 4  
[Source: docs/epics/epic-4-output-manifest-v02.md — Story 4.3]

- Load existing manifest if present.  
- Append new entry.  
- Write manifest atomically.  
- Ensure valid JSON schema.

## Acceptance Criteria

### AC1: Manifest Loading & Validation
- ✅ Implement `ManifestManager` (new module `pdf2llm/manifest.py`) with methods: `load(path: Path) -> ManifestData` and `write(manifest: ManifestData, path: Path)`.  
- ✅ Handles missing file by returning empty manifest structure.  
- ✅ Validates JSON schema (required keys) and raises descriptive error on corruption.

### AC2: Entry Upsert
- ✅ Provide `register_document(manifest: ManifestData, entry: dict) -> ManifestData` that adds or replaces document entry by `doc_id`.  
- ✅ Maintains deterministic ordering (e.g., sort by slug).  
- ✅ Tracks `chunk_count`, `pages` (if available), `strategy` (parser/formatter info).

### AC3: Atomic Write & Timestamp
- ✅ `write` uses atomic rename (`atomic_write`) to avoid partial files.  
- ✅ Updates manifest-level `updated_utc` timestamp.  
- ✅ Preserves previous `created_utc` when manifest already exists.

---

## Implementation Guidance

1. **Data Structures:** Define `ManifestData` dataclass or dict with keys: `created_utc`, `updated_utc`, `docs` (list of entries).  
2. **Serialization:** Use `json` module with `ensure_ascii=False` and sorted keys.  
3. **Ordering:** After upsert, sort entries by slug to support diff-friendly output.  
4. **Integration:** Pipeline/writer will call `register_document` after writing outputs (Story 4.4).
