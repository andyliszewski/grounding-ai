## Story Description

As a **developer**, I want to **compute deterministic hashes for documents and chunks** so that manifests, deduplication workflows, and downstream consumers can rely on stable identifiers for content integrity checks.

This story rounds out Epic 3 by introducing a hashing utility that favors BLAKE3 for speed (with SHA-256 fallback when BLAKE3 unavailable). It hashes both full Markdown documents and chunk bodies, ensuring metadata from Story 3.3 can embed trustworthy fingerprints.

---

## Context

### From PRD v0.2  
[Source: docs/prd.md — Sections "Integrity & Auditing" & "Determinism"]

- Each chunk and document must include a reproducible hash for integrity validation.  
- Hashing must exclude front matter to avoid double counting metadata.  
- When algorithms change, fallbacks should keep the system working without major rewrites.

### From Architecture v0.2  
[Source: docs/architecture.md — Sections 5 File Context & 7 Chunking]

- Hash helpers should live in a shared module and be reusable by pipeline, chunk metadata, and future manifest writers.  
- Prefer BLAKE3 for performance but support SHA-256 when BLAKE3 is unavailable in the runtime environment.  
- Hash results stored with `FileContext` and `ChunkMetadata`.

### From Epic 3  
[Source: docs/epics/epic-3-chunking-metadata-v02.md — Story 3.4]

- Hash chunk text content and full documents.  
- Exclude YAML front matter from chunk hashing.  
- Ensure determinism across runs.

## Acceptance Criteria

### AC1: Hashing Utilities
- ✅ Expose `hash_content(text: str, algorithm: Literal["blake3", "sha256"]="blake3") -> str`.  
- ✅ Provide `hash_document(markdown: str) -> dict[str, str]` returning both BLAKE3 and SHA-256 hex digests.  
- ✅ Gracefully fallback to SHA-256 when BLAKE3 dependency missing (log once).

### AC2: Chunk Hash Integration
- ✅ Helper `hash_chunk(chunk_text: str) -> str` hashes body-only content (caller strips YAML).  
- ✅ Function accepts `skip_front_matter: bool = True` default to strip YAML before hashing.  
- ✅ Deterministic output verified with repeated calls.

### AC3: Pipeline & Metadata Wiring
- ✅ Pipeline stores document hash map on `FileContext` for manifest use.  
- ✅ Story 3.3 metadata uses chunk hashes produced here.  
- ✅ Hash computations recorded in stats/logs (optional debug-level) without degrading performance.

---

## Implementation Guidance

1. **Dependency:** Add `blake3>=0.4.1` as optional dependency; handle ImportError gracefully.  
2. **Module:** Extend `pdf2llm/hashing.py` (introduced in Story 3.2) with chunk/document helpers.  
3. **Front Matter Removal:** Implement utility to strip YAML header (`---` … `---`) before hashing chunks.  
4. **Caching:** Consider memoizing algorithm availability check to avoid repeated ImportError handling.  
5. **Logging:** Emit a single warning when falling back to SHA-256 due to missing BLAKE3.
