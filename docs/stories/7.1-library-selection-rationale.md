# Story 7.1: OMR Library Selection Rationale

**Created:** 2025-10-18
**Story:** 7.1 - OMR Library Research & Selection
**Decision:** Audiveris (Primary) / homr (Fallback)

---

## Executive Summary

After comprehensive research and evaluation of 5 OMR libraries, **Audiveris** is selected as the primary OMR engine for pdf2llm Epic 7, with **homr** as a fallback option.

**Key Decision Factor:** Accuracy over ease of implementation. Music education use case demands 85-90% accuracy (Audiveris) over 70-80% (Python-native alternatives). Integration complexity is a one-time cost; accuracy issues are ongoing pain.

---

## Selection Decision

### Primary: **Audiveris**

**Rationale:**
1. **Superior Accuracy (85-90%):** 15-25% fewer errors than alternatives
2. **Production-Proven:** Used in real-world music digitization (IMSLP community)
3. **Mature & Stable:** 2.1k stars, 8 years development, comprehensive documentation
4. **Rich Output:** MusicXML 4.0 + .omr format with full OMR data exposure
5. **Local-Only:** No network calls, privacy-compliant
6. **Active Community:** Responsive maintainers, frequent releases

**Trade-offs Accepted:**
- JRE 24+ dependency (~200MB)
- Subprocess integration (vs native Python)
- AGPL-3.0 license (vs MIT)

**Integration Strategy:**
- **Phase 1 (Story 7.1):** Subprocess integration via CLI
- **Phase 2 (Story 7.2):** Evaluate py4j if API control needed
- Abstract behind parser interface for future flexibility

---

### Fallback: **homr**

**When to Use:**
- Audiveris PoC fails (JRE issues, platform incompatibility)
- Python-native becomes hard requirement
- MIT licensing critical for commercial use
- Photo-quality input primary use case

**Advantages:**
- Easy installation (pip install homr)
- MIT license (permissive)
- Better photo handling
- Native Python

**Trade-offs:**
- ~10-15% lower accuracy (75-80%)
- Smaller community, less documentation
- Newer, less mature

---

## Decision Framework

### Epic 7 Requirements (Prioritized)

1. **Accuracy** (Highest Priority)
   - Target: >80% for common Western notation
   - **Audiveris:** ✅ 85-90%
   - **homr:** ⚠️ 75-80%
   - **oemer:** ⚠️ 70-75%

2. **Local-Only Processing** (Hard Requirement)
   - **All candidates:** ✅ Met

3. **Python >=3.10 Compatible** (Hard Requirement)
   - **Audiveris:** ✅ Subprocess integration
   - **homr/oemer:** ✅ Native Python

4. **Open-Source, Commercial-Friendly**
   - **Audiveris:** ⚠️ AGPL-3.0 (acceptable for CLI tool)
   - **homr:** ✅ MIT
   - **oemer:** ✅ MIT

5. **Active Maintenance** (Important)
   - **Audiveris:** ✅ Excellent (Sep 2025 release)
   - **homr:** ✅ Good (2024 active)
   - **oemer:** ⚠️ Moderate (points to homr)

6. **Integration Complexity** (Lower Priority)
   - **Audiveris:** ⚠️ Moderate (JRE + subprocess)
   - **homr/oemer:** ✅ Easy (pip install)

**Result:** Audiveris scores highest on critical factors (accuracy, maturity), acceptable on others.

## Risk Assessment

### Audiveris Risks

**Risk 1: JRE Installation Complexity**
- **Impact:** Medium (users need to install JRE 24+)
- **Likelihood:** Low (installers available for all platforms)
- **Mitigation:**
  - Provide detailed installation guide (macOS, Linux, Windows)
  - Test installation on all platforms
  - Document Homebrew (macOS), apt (Linux), installer (Windows)
  - Fallback to homr if installation fails

**Risk 2: Subprocess Integration Less Robust**
- **Impact:** Low (subprocess is proven pattern)
- **Likelihood:** Low (many projects use subprocess for Java tools)
- **Mitigation:**
  - Abstract behind OMR parser interface
  - Implement robust error handling
  - Log stderr/stdout for debugging
  - Evaluate py4j in Story 7.2 if needed

**Risk 3: AGPL Licensing Implications**
- **Impact:** Low (CLI tool, no network service)
- **Likelihood:** Very Low (AGPL network provisions don't apply)
- **Mitigation:**
  - Document licensing clearly
  - Confirm with legal if commercial use planned
  - Note commercial exception exists
  - Fallback to homr (MIT) if needed

**Risk 4: Accuracy Claims Unverified**
- **Impact:** High (if actual accuracy <80%, fails use case)
- **Likelihood:** Low (large user base, IMSLP usage validates)
- **Mitigation:**
  - **PoC testing** (Story 7.1): Test with real music notation PDFs
  - Validate MusicXML output with music21
  - Manual inspection of accuracy
  - If accuracy <80%, switch to homr

## Alternative Scenarios

### Scenario 1: Audiveris PoC Fails
**Trigger:** PoC accuracy <75% OR installation issues

**Action:**
1. Switch to homr as primary
2. Update Epic 7 stories to reflect homr integration
3. Document decision and rationale

**Impact:**
- Easier integration (native Python)
- Possibly lower accuracy (75-80% vs 85-90%)
- MIT license (cleaner)

### Scenario 2: Licensing Becomes Blocker
**Trigger:** Commercial partnership requires MIT/Apache licensing

**Action:**
1. Switch to homr (MIT) or oemer (MIT)
2. Accept ~10-15% accuracy reduction
3. Document trade-off decision

### Scenario 3: Python-Native Becomes Hard Requirement
**Trigger:** Architecture change requires in-process OMR

**Action:**
1. Evaluate py4j integration (keep Audiveris)
2. If py4j too complex, switch to homr
3. Update Story 7.2 implementation
