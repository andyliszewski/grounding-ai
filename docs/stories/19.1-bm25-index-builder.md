## Story

**As a** grounding-ai contributor,
**I want** per-agent BM25 indexes built alongside the existing FAISS indexes, with the same incremental and tombstone semantics,
**so that** Story 19.2 has a clean, pure retrieval surface to fuse with the dense channel, and so the index storage format is versioned and migration-safe from day one.

---

## Acceptance Criteria

1. A new module `grounding/bm25.py` exports:
   - `tokenize(text: str) -> list[str]` — simple lowercase `\w+` tokenizer.
   - `BM25Index` class or equivalent — wraps `rank_bm25.BM25Okapi`.
   - `write_bm25_index(chunk_bodies, chunk_ids, output_dir)` — builds and serializes.
   - `load_bm25_index(output_dir) -> BM25Index | None` — returns `None` (not raises) if files absent.
   - `append_to_bm25_index(new_bodies, new_ids, output_dir)` — incremental add.
   - `tombstone_bm25_documents(doc_ids, output_dir)` — soft-delete parity with FAISS.
   - `search_bm25(index, query_tokens, top_k)` — returns `list[tuple[chunk_id, rank, score]]`, tombstone-filtered.
2. Serialization format:
   - `_bm25.pkl` — pickled `BM25Okapi` state (tokenized corpus + internal arrays).
   - `_bm25_map.json` — metadata and chunk map, with schema versioning.
3. `_bm25_map.json` schema (v1):
   ```json
   {
     "format_version": 1,
     "tokenizer": "whitespace_lowercase_v1",
     "rank_bm25_version": "<pkg_version>",
     "total": 5,
     "tombstone_count": 0,
     "created_utc": "...",
     "updated_utc": "...",
     "chunks": [
       {"bm25_index": 0, "chunk_id": "...", "doc_id": "...", "deleted_utc": null}
     ]
   }
   ```
4. Loader rejects files whose `tokenizer` field does not match the current tokenizer identity. Rejection raises `BM25FormatError` with an actionable message ("rebuild with `grounding embeddings --agent <name>`").
5. `grounding embeddings` command (`grounding/cli.py:embeddings_command`) is extended so that every full-build or `--incremental` run produces/updates `_bm25.pkl` and `_bm25_map.json` alongside the existing FAISS artifacts. `--incremental` behavior:
   - New chunks → `append_to_bm25_index(...)`.
   - Updated doc_ids → `tombstone_bm25_documents(list(updated_doc_ids), ...)` before the append (mirrors `tombstone_documents` call at `cli.py:443`).
   - Empty corpus → no-op, no error.
6. Full-build path writes a fresh `_bm25.pkl` + `_bm25_map.json` (analogous to `write_vector_index` at `cli.py:449`).
7. Empty-corpus and single-chunk-corpus are handled without exceptions from either path.
8. Tombstone semantics parity with FAISS:
   - `search_bm25` filters tombstoned entries by default.
   - Tombstone count is tracked in `_bm25_map.json` and logged analogously to the FAISS tombstone warning.
9. Unit tests in `tests/test_bm25.py` cover: build + save + load round-trip, tokenizer identity guard, incremental append preserves ordering, tombstone filtering in search, empty corpus, single-chunk corpus, tokenizer output on representative input (code identifiers, punctuation, mixed case). No network.
10. Integration: after `grounding embeddings --agent <mini-test> ...` the mini-corpus fixture directory gains committed `_bm25.pkl` and `_bm25_map.json`; one integration test asserts both files exist and are loadable. The `tests/eval_fixtures/mini_index/` is updated in this PR.
11. New runtime dependency `rank_bm25>=0.2.2` added to `pyproject.toml`; no other new runtime deps.
12. No change to the FAISS index format, the `_chunk_map.json` schema, or any existing test. BM25 artifacts are purely additive.

### Why re-tokenize + rebuild on append (not mutate state)

`rank_bm25.BM25Okapi` computes IDF in `__init__` from the full tokenized corpus. Mutating internal arrays in place is not part of its public API. Rebuilding on append is O(total chunks) in tokenization, O(total chunks × avg_tokens) in IDF computation — on 100k chunks at a few KB each, that's still a few seconds. Acceptable since (a) `--incremental` typically adds tens of chunks at a time, (b) BM25 state pickle is small, (c) the alternative (vendoring and patching `rank_bm25`) is worse.

If profiling ever shows this is a real bottleneck, incrementalize IDF on a vendored fork. Not a Day 1 concern.

### Parallel-array contract with FAISS

FAISS integer doc ids are assigned in insertion order. This story ensures `write_bm25_index` and `append_to_bm25_index` receive chunk bodies and ids in the same order `write_vector_index` / `append_to_vector_index` receive their metadata. That means:

```
faiss_internal_id[i] ↔ chunk_ids[i] ↔ bm25_internal_id[i]  (for all i < total_before_tombstones)
```

Story 19.2's RRF merge exploits this — both channels produce `chunk_id` handles and 1-indexed ranks, and the merge is a pure Python dict lookup. No chunk-id remapping needed.

### `chunk_bodies` vs `chunk_metadata`

Looking at `cli.py` around lines 430–450, the loop builds a `chunk_metadata` list of dicts (one per chunk, used by `append_to_vector_index`). The chunk *bodies* (raw text used for tokenization) are available in the same loop iteration — confirm this during implementation and plumb a parallel `chunk_bodies: list[str]` list through the build. If the existing code path doesn't retain bodies by the time it reaches the write step, a minor restructure may be needed; flag as a risk.

### Atomic write for pickle

`grounding/utils.py:atomic_write` targets text content. For the pickle, either:
1. Extend `atomic_write` with a bytes mode.
2. Inline the temp-file-then-rename pattern locally in `bm25.py`.

Recommend option 2 to keep `bm25.py` self-contained and `utils.py` unchanged. ~10 lines.

### Empty-corpus shape

`BM25Okapi([])` raises in some `rank_bm25` versions. Gate on empty input in `write_bm25_index`: write `_bm25.pkl` as a pickled `None` sentinel (or skip the pickle and mark `total: 0` in the map); document the chosen shape; have `load_bm25_index` return `None` or an empty-shaped index consistently. Unit tests cover both paths.

### Tokenizer identity (future-proofing)

`TOKENIZER_IDENTITY = "whitespace_lowercase_v1"` is the contract. If anyone ever adds stemming, it becomes `"porter_v1"` or similar — and old pickles are rejected with a clear rebuild message rather than silently producing wrong scores. This is cheap insurance now; expensive retrofit later.

### Out of scope

- Tokenizer customization via config — defer until measurement justifies it.
- Hybrid retrieval / RRF merge — Story 19.2.
- CLI flag / MCP / `local_rag.py` wiring — Story 19.3.
- Performance optimization beyond "acceptable on realistic corpora" — Story 19.4 may revisit if numbers warrant.

## QA Results

### Review Date: 2026-04-15

### Reviewed By: Quinn (Test Architect)

### Code Quality Assessment

Clean, self-contained module. `grounding/bm25.py` is a single-file
implementation that mirrors the FAISS sidecar patterns
(`vector_store.py`) almost one-for-one: filter-at-search-time tombstones,
temp-file-then-rename atomic writes, fetch-multiplier to absorb
tombstone loss. The parallel-array contract with FAISS is visibly
preserved end-to-end — `ordered_ids = list(embeddings.keys())` in
`cli.py:448` feeds both `write_vector_index` and `write_bm25_index`, so
bm25 internal id `i` and FAISS integer id `i` refer to the same
`chunk_id`. 19.2's RRF merge reduces to a dict lookup.

Two implementation choices deserve explicit endorsement:

1. **Empty-corpus sentinel** — pickle `None`, map with `total: 0`.
   Distinguishes "index exists but empty" from "index absent" (loader
   returns `None` only for absent). Matches dev-notes guidance.
2. **`_grounding_tokenized_corpus` namespaced attribute** — `rank_bm25.BM25Okapi`
   consumes its tokenized input at `__init__` without retaining it, so
   the dev stashes it on the instance under a prefixed attr that
   round-trips through pickle. Clean solution to a real pain point; the
   append path would otherwise require either (a) vendoring rank_bm25
   or (b) storing the tokenized corpus in the JSON map (which would
   bloat it 3-5×). The chosen approach keeps the map small and the
   module dependency-minimal. Old pickles without the attr fail with a
   clear `BM25FormatError` + rebuild hint — fail-loud is correct here.

### Refactoring Performed

None. Implementation is already at the right level of abstraction for
Day 1 of Epic 19. Premature polish would be scope creep.

`json.load` over a trusted file, pickle load from a path under the

the file is written and consumed by this same process tree. No new
dependency vulnerabilities (`rank_bm25` is pure-Python, numpy-only).
