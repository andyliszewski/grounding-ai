## Story

**As a** grounding-ai user,
**I want** the two-stage retrieval flow (FAISS top-N тЖТ cross-encoder rerank тЖТ top-k) to be available in every retrieval surface,
**so that** the reranker shipped in 18.1 actually runs on my queries through the agentic tool, MCP server, and local RAG CLI, with opt-in behavior and zero regression when disabled.

---

## Acceptance Criteria

1. `SearchCorpusTool` (`scripts/search_corpus_tool.py`) accepts an optional `rerank_config: RerankConfig | None` as a constructor kwarg. When `rerank_config is None` or `rerank_config.enabled is False`, behavior is bit-for-bit identical to the pre-epic flow.
2. When reranking is enabled on `SearchCorpusTool`: the first stage fetches `max(rerank_config.pool_size, top_k)` results from FAISS, the second stage calls `grounding.reranker.rerank(query, results, config=rerank_config)`, the final output is the top-`top_k` of the reranked list.
3. Rank numbering is re-assigned 1-indexed after reranking; `faiss_distance` is preserved on every returned result dict (per 18.1 contract).
4. MCP server (`mcp_servers/corpus_search/server.py`):
   - `search_corpus(...)` function signature accepts an optional `rerank_config: RerankConfig | None` parameter.
   - `call_tool("search_corpus", arguments)` reads optional `rerank_enabled: bool`, `rerank_pool_size: int`, `rerank_model: str` arguments; when `rerank_enabled is True` it constructs a `RerankConfig(enabled=True, model=<arg or default>, pool_size=<arg or default>)` and threads it through.
   - When absent, behavior is unchanged.
5. Local RAG (`scripts/local_rag.py`):
   - The retrieval helper that `format_context` consumes accepts a `rerank_config` argument.
   - When enabled, the two-stage flow runs before citation prefixing; when disabled, behavior is unchanged.
6. Citation prefix (Epic 17) continues to reflect the **re-ranked** top chunk тАФ i.e., prefix formatting runs after reranking, not before.
7. When `rerank_config.enabled is True` but the FAISS stage returns fewer than `pool_size` results (small corpus), reranking runs on whatever was returned; no error, no warning spam.
8. Top-k clamping (1..20) still applies to the final output after reranking; pool size is not clamped (user can legitimately want pool_size=100).
9. Unit tests extend the existing `tests/test_search_corpus_tool.py` and `tests/test_mcp_server_output.py`, and add `tests/test_local_rag_rerank.py` as needed, to cover: rerank-on changes ordering vs rerank-off, rerank-off is bit-for-bit identical to current behavior, small-pool graceful behavior, citation prefix reflects reranked top chunk.
10. Epic 16 mini-corpus eval gate stays green on this PR (rerank is off by default; 18.4 is where the measured flip happens).
11. No new runtime dependencies beyond what 18.1 introduced (which was also zero).

### Pool-size vs top-k semantics

- `top_k` тАФ what the user wants back (clamped 1тАУ20).
- `pool_size` тАФ how many candidates FAISS fetches to feed the reranker. Default 50.
- `fetch_k = max(pool_size, top_k)` тАФ so a user asking for `top_k=30` (if we ever raise the clamp) still gets at least 30 candidates before rerank.
- When rerank is disabled: `fetch_k = top_k` (unchanged).

### Re-rank + citation interaction

Citation prefix (from 17.3) is built in `_format_results`. Reranking happens before formatting. The prefix consequently reflects the reranked top chunk's `source`, `page_start`, `page_end`, `section_heading`. No change required in the citation helper тАФ the dict fields are preserved through reranking per 18.1's AC#4.

### FAISS distance vs rerank score in the formatted output

18.1 preserves the original FAISS distance under `faiss_distance` and overwrites `score` with the rerank score. `_format_results` reads neither field directly (current implementation), so the formatted output doesn't need to change. If a future story wants to surface both in the output, that's a separate change; out of scope here.

### Opt-in default

All three surfaces receive a `rerank_config` that is `None` (or `enabled=False`) by default in this story. Story 18.3 adds the user-facing flags. Story 18.4 measures and potentially flips the default. Keeping 18.2 purely plumbing (no user-visible default change) means this PR is safe to merge behind the rest.

### Out of scope

- CLI flags, MCP schema documentation beyond the minimum required for tool calls to work, config.yaml plumbing тАФ Story 18.3.
- Measuring the lift, baseline refresh, flipping the default тАФ Story 18.4.
- New citation-surfacing features (e.g., showing both scores in output) тАФ separate story.
- Performance benchmarking and latency optimization тАФ Story 18.4 may touch this.
