## Dev Notes

### Story Context

This is the foundational research story for Epic 7 (OMR Support). The goal is to select the most suitable OMR library for extracting music notation from PDF documents. Based on Epic 7 decisions (made during PO consultation), **Audiveris** has been pre-selected as the primary choice due to its superior accuracy (85-90% vs oemer's 70-75%). However, this story includes validation of that decision through hands-on research and proof-of-concept work.

### Project Background

pdf2llm is a Python CLI tool (Python >=3.10) that converts PDFs to LLM-ready Markdown artifacts. The project follows these principles:
- **Local-only**: No network calls, privacy-focused
- **Determinism**: Same inputs → identical outputs
- **Open-source**: All dependencies must be open-source compatible
- **Minimal dependencies**: Keep the dependency footprint reasonable

[Source: docs/architecture.md, CLAUDE.md]

### OMR Library Decision Framework

The selection criteria prioritize (in order):
1. **Accuracy**: Music notation extraction accuracy for common Western notation
2. **Local-only**: Must work entirely offline
3. **Python compatibility**: Must integrate with Python >=3.10
4. **Licensing**: Open-source, commercial-friendly
5. **Maintenance**: Active development or stable mature project
6. **Integration complexity**: Reasonable integration effort (but accuracy is more important)

[Source: docs/epics/epic-7-omr-support-v02.md - OMR Library Selection]

### Audiveris (Primary Candidate - PRE-SELECTED)

**Why pre-selected**: Music education use case demands accuracy over ease of implementation. Setup complexity is a one-time cost; accuracy issues are ongoing pain.

**Key Characteristics**:
- **Type**: Java-based desktop application with OMR engine
- **Accuracy**: 85-90% for common Western notation
- **Integration**: Python via py4j (Java bridge) or JPype
- **Dependencies**: Java Runtime Environment (JRE) >=11
- **Size**: ~200MB installed
- **License**: AGPL v3 (with potential commercial exception)
- **Status**: Mature, actively maintained
- **Repository**: https://github.com/Audiveris/audiveris

**Integration Approach**:
- Use py4j to bridge Python → Java Audiveris engine
- Audiveris processes PDF → MusicXML output
- Python code wraps Audiveris CLI or Java API calls

**Validation Needed**:
- Confirm accuracy claims with test PDFs
- Verify Python integration feasibility
- Document installation process
- Test on macOS (primary dev environment)

### oemer (Fallback Candidate)

**Characteristics**:
- **Type**: Python-native deep learning OMR
- **Accuracy**: 70-75% typical
- **Integration**: Direct Python package installation
- **Dependencies**: PyTorch (~500MB), cv2, numpy
- **License**: MIT or Apache 2.0 (likely)
- **Status**: Actively developed, research-oriented
- **Repository**: https://github.com/BreezeWhite/oemer

**Fallback Trigger**: Use oemer if Audiveris Java integration proves problematic during PoC.

### music21 Library

For PoC validation and future use:
- **Purpose**: Music notation analysis and validation
- **License**: BSD (open-source compatible)
- **Python Support**: Python >=3.10 compatible
- **Use**: Validate MusicXML output, extract musical features (Story 7.4)

[Source: docs/epics/epic-7-omr-support-v02.md - Dependencies]

### Test Data Requirements

For PoC testing, need sample music notation PDFs:
1. **Simple melody**: Single-staff, treble clef, C major, quarter/half notes
2. **Piano score** (optional for PoC): Grand staff (treble + bass)

Suggested sources:
- Public domain sheet music (IMSLP.org)
- Generated test scores from MuseScore
- Music theory textbook examples

### Success Criteria for PoC

The PoC is successful if:
- Audiveris processes a simple melody PDF without crashing
- MusicXML output is generated
- Output validates against MusicXML schema (music21 can parse it)
- Manual inspection shows >80% notation accuracy (notes, clefs, key signature)

If PoC fails:
- Document failure mode
- Attempt oemer PoC as fallback
- Report findings to inform Story 7.2 implementation decisions

This story is research-only. No changes to pdf2llm architecture or code are made. Architecture updates happen in Story 7.2 (Parser Integration).
