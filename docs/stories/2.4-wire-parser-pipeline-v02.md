## Story Description

As a **developer**, I want to **connect the scanner, parser, and formatter into the core pipeline** so that the CLI can process an input directory end-to-end and emit Markdown artifacts for each PDF with deterministic behavior.

This story stitches together the modules delivered in Stories 2.1–2.3, ensures shared state (file context, metrics) flows correctly, and exposes OCR and output options via the pipeline interface. It prepares the groundwork for CLI integration and manifest generation.

---

## Context

### From PRD v0.2  
[Source: docs/prd.md — Sections "Batch Processing" & "Output Guarantees"]

- Users expect `pdf2llm` to take an input folder and produce Markdown outputs automatically.  
- Batch runs must continue on individual file failures while logging the details.  
- Outputs need to be deterministic across reruns to support diff-based QA.

### From Architecture v0.2  
[Source: docs/architecture.md — Sections 3 Pipeline Components & 5 File Context]

- `pipeline.py` orchestrates scanning → parsing → formatting → writing.  
- File context tracks source path, output directory, SHA-1 hash, status, and metadata.  
- Progress logging and stats aggregation live at the pipeline layer.

### From Epic 2  
[Source: docs/epics/epic-2-parser-integration-v02.md — Integration Notes & Acceptance Criteria]

- Scanner, parser, and formatter must be wired together with proper error handling and logging.  
- OCR mode should default to `auto` but respect user override.  
- Individual failures should not abort the batch; results must include success/failure summary.

## Acceptance Criteria

### AC1: Pipeline Composition
- ✅ `pipeline.py` exposes a function (e.g., `run_pipeline(config: PipelineConfig)`) that:  
  - Invokes `scan_pdfs` to get ordered list of PDFs.  
  - For each PDF, invokes `parse_pdf` and `format_markdown`.  
  - Collects results into a context object for the writer to consume.

### AC2: Error Isolation
- ✅ Failures in parsing or formatting capture the error, log with context, and continue processing remaining PDFs.  
- ✅ Pipeline aggregates success/failure counts and exposes them via return value or stats object.

### AC3: OCR & Configuration Handling
- ✅ Pipeline accepts OCR mode argument (`auto/on/off`) and forwards it to `parse_pdf`.  
- ✅ Optional formatter fallback flag threaded through for future stories.  
- ✅ Configuration validated (e.g., unknown OCR mode blocked at pipeline edge).

### AC4: Logging & Metrics
- ✅ For each processed file, emit structured log (slug, OCR mode, elapsed times).  
- ✅ Summary log at end (total processed, succeeded, failed, skipped).  
- ✅ Stats module updated to include parser/formatter timings.

### AC5: Tests
- ✅ Pipeline unit/integration test uses fixtures to assert ordering, error isolation, and summary stats.  
- ✅ Tests cover scenario where one PDF raises `ParseError` and others succeed.  
- ✅ Tests verify OCR mode propagated correctly to parser mock.

---

## Implementation Guidance

1. **Define Config/Data Structures:**  
   - Extend `pipeline.PipelineConfig` (or create) with fields: `input_dir`, `output_dir`, `ocr_mode`, `allow_plaintext_fallback`.  
   - Introduce `FileContext` namedtuple/dataclass capturing per-file state (path, slug, hash, status, error message).

2. **Integrate Modules:**  
   - Import `scan_pdfs`, `parse_pdf`, `format_markdown`.  
   - Iterate scanner output; for each file:  
     - Build context (compute slug + Hash via `stats` or use placeholder).  
     - Try parse → format; on success, stash Markdown on context for writer.  
     - On failure, mark context as failed/skipped.

3. **Hook Into Writer:**  
   - For MVP, call existing `writer.write_markdown` (if available) or stub call to be filled in next epic.  
   - Ensure output directory prepared via `utils.ensure_dir`.

4. **Metrics & Logging:**  
   - Use `logging_setup.get_logger("pdf2llm.pipeline")`.  
   - Record per-file durations using `time.perf_counter`.  
   - Update `stats` module with new counters (`parsed_count`, `formatted_count`, `failed_count`).

5. **Testing Strategy:**  
   - Mock parser/formatter to inject success/failure.  
   - Validate deterministic order of operations by capturing call sequence.  
   - Assert summary object matches expected counts.
