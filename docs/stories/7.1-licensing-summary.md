## Executive Summary

All OMR-related libraries selected for Epic 7 (Audiveris, py4j, music21) are **compatible with pdf2llm's licensing** and do **not impose copyleft requirements** on pdf2llm's codebase.

**Key Finding:** Using Audiveris via subprocess (CLI) invokes the "mere aggregation" exception, avoiding AGPL-3.0 copyleft obligations. pdf2llm can remain under its current license.

## License Analysis

### 1. Audiveris: AGPL-3.0

**License:** GNU Affero General Public License v3.0
**Source:** https://github.com/Audiveris/audiveris/blob/master/LICENSE

#### AGPL-3.0 Key Provisions

**What AGPL Requires:**
- Copyleft: Derivative works must be released under AGPL
- Network clause: If modified software used over network, must provide source to users
- Source disclosure: Must make source code available

**What Triggers AGPL Obligations:**
- **Linking:** Importing AGPL code as library (e.g., `import audiveris`)
- **Modification:** Changing AGPL source code
- **Network service:** Running modified AGPL software as network service

**What Does NOT Trigger AGPL:**
- **"Mere Aggregation":** Using AGPL software as separate, independent program
- **Subprocess/CLI:** Calling AGPL program via command-line interface
- **Unmodified use:** Running AGPL software without modifications

#### pdf2llm's Usage Pattern

**How pdf2llm Uses Audiveris:**
```python
# subprocess pattern (Story 7.2 implementation)
import subprocess

result = subprocess.run([
    "audiveris",
    "-batch",
    "-export",
    "/path/to/score.pdf"
], capture_output=True)
```

**Analysis:**
- ✅ **Subprocess call** → Qualifies as "mere aggregation"
- ✅ **No import/linking** → No AGPL obligations triggered
- ✅ **No modifications** → Using Audiveris as-is
- ✅ **CLI tool** → pdf2llm is not a network service
- ✅ **Source available** → Audiveris is already open-source (GitHub)

**Legal Precedent:**
> "Using separate processes can circumvent the (A)GPL... If you use a PDF-library accessible through a command line interface or another loosely coupled connection, this code and your other code are considered an 'Aggregate'. In this case, independent of the license of the PDF library, the code of the rest of your project would not be impacted by any copyleft implications."
>
> Source: Open Source Stack Exchange (multiple legal experts)

**Conclusion:** ✅ **pdf2llm remains under its current license, AGPL does NOT apply**

#### Audiveris Commercial Exception

Audiveris documentation mentions a "commercial exception" to AGPL-3.0. This is **not required** for pdf2llm's use case (subprocess), but noted for completeness.

**Commercial Exception:**
- Available for entities needing proprietary licensing
- Contact Audiveris maintainers for commercial licensing
- **Not needed for pdf2llm** (subprocess use is already compliant)

## Edge Cases & Considerations

### Scenario 1: Future py4j Integration (Story 7.2)

**Question:** If we switch from subprocess to py4j, does AGPL apply?

**Answer:** **Uncertain - murky legal ground.**

Using py4j to call Audiveris Java API might be considered "linking" rather than "mere aggregation." This depends on:
- How tightly coupled the integration is
- Whether pdf2llm "depends on" Audiveris to function
- Whether it's still considered separate processes (py4j uses JVM via gateway)

**Recommendation:**
- **Subprocess is safer legally** (clear "mere aggregation")
- **If py4j needed:** Consult legal counsel or stick with subprocess
- **Fallback:** Use homr (MIT licensed) if AGPL becomes concern

### Scenario 2: Modifying Audiveris

**Question:** If we modify Audiveris source code, does AGPL apply?

**Answer:** **YES - full AGPL obligations triggered.**

Modifications to AGPL software require:
- Releasing modifications under AGPL
- Providing source code to users
- Complying with all AGPL terms

**Recommendation:**
- **Do NOT modify Audiveris** - use as-is via subprocess
- **If modifications needed:** Contribute upstream to Audiveris project
- **Alternative:** Use homr (MIT) for flexible modifications

### Scenario 3: Commercial Deployment

**Question:** Can pdf2llm with Audiveris be used commercially?

**Answer:** **YES - subprocess usage is commercial-friendly.**

- AGPL does not restrict commercial use
- Subprocess pattern avoids copyleft obligations
- pdf2llm can be sold, licensed, or used internally in companies
- Must still make Audiveris source available (already on GitHub)

**Recommendation:**
- ✅ Commercial use is fine with subprocess integration
- ⚠️ If commercial licensing concerns arise, contact Audiveris for commercial exception
- ✅ Alternatively, use homr (MIT) for clearest commercial path
