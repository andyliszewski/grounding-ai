## Story

**As a** grounding-ai maintainer,
**I want** the eval harness to run automatically on PRs that touch retrieval and to gate against a committed baseline,
**so that** silent retrieval regressions are caught before merge and Tier 1 roadmap improvements have a credible measurement against published numbers.

---

## Acceptance Criteria

1. A GitHub Actions workflow at `.github/workflows/eval.yml` runs `grounding eval` against the **mini test corpus** (from Story 16.2) on every PR that modifies any of: `grounding/embedder.py`, `grounding/vector_store.py`, `grounding/chunker.py`, `grounding/cli.py`, `grounding/manifest.py`, `grounding/agent_filter.py`, `grounding/eval/**`, `tests/eval_fixtures/**`, `docs/eval/**`, `pyproject.toml`.
2. The workflow installs project dependencies, runs the eval, and gates the PR using `--baseline docs/eval/baselines/mini-corpus.json --fail-under 0.05`.
3. A baseline file `docs/eval/baselines/mini-corpus.json` is committed to the repo, generated from the mini corpus at a recorded UTC timestamp and fixture version.
4. The workflow's total runtime stays under 5 minutes (target: under 2 minutes), achieved via dependency caching and pre-built mini index (committed or built once and cached).
5. On regression (exit code 1), the workflow fails the PR check and posts a comment containing the eval Markdown report. On success, no comment is posted (or a collapsed one).
6. `docs/eval/README.md` documents: how to add fixtures, how to run the harness locally, how to refresh the baseline (including the explicit "human review required" step), how the CI gate works, and what each exit code means.
7. A demonstration PR (or recorded link in epic notes) shows the gate firing on a deliberate regression — e.g., chunker chunk_size set to 100 — proving end-to-end behavior. The demo PR is closed without merge.
8. ROADMAP.md Tier 1 #1 is moved from "next up" to "Shipped" in the same PR that merges this story; Tier 1 #2 is explicitly called out as unblocked.

## Dev Notes

### Three-repo layering (document in README)

grounding-ai is one of three coordinated repos:

| Repo | Visibility | What lives there | Eval implication |
|------|------------|------------------|------------------|
| `grounding-ai` | Public | Pipeline code, schema, mini test corpus, schema-example fixture, mini baseline, CI workflow | CI gate runs here against the mini corpus only |
| `grounding-ai-private` | Private | Real corpus content, baselines for real agents | Real-agent fixtures and baselines live here; eval is run locally or in private CI |
| `my-agents` | Private | The maintainer's actual agent YAMLs (e.g., `data-scientist.yaml`) | Real-agent fixtures reference these; never copied into public repo |

Public-repo contributors ship retrieval *code*. Real-agent eval is the maintainer's private concern. The public CI gate guards against retrieval-code regressions; private eval runs guard against corpus-quality regressions. Document this distinction explicitly in `docs/eval/README.md` so external contributors understand what the public eval does and does not cover.

### Why Mini Corpus, Not Real Agent Corpora

CI cannot ship real corpora (size, licensing, privacy). The mini corpus from Story 16.2 (3-5 small chunks committed under `tests/eval_fixtures/`) is the only viable target for public CI. Real baselines live outside this repo per the layering above.

### Workflow Skeleton (target)

```yaml
name: eval
on:
  pull_request:
    paths:
      - 'grounding/embedder.py'
      - 'grounding/vector_store.py'
      - 'grounding/chunker.py'
      - 'grounding/agent_filter.py'
      - 'grounding/eval/**'
      - 'tests/eval_fixtures/**'
      - 'docs/eval/**'

permissions:
  pull-requests: write
  contents: read

jobs:
  eval:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.13'
          cache: 'pip'
      - name: Cache sentence-transformers model
        uses: actions/cache@v4
        with:
          path: |
            ~/.cache/torch/sentence_transformers
            ~/.cache/huggingface
          key: st-${{ runner.os }}-all-MiniLM-L6-v2-v1
      - name: Install
        run: pip install -e .
      - name: Build mini index
        run: grounding embeddings --corpus tests/eval_fixtures/mini_corpus --out tests/eval_fixtures/mini_index
      - name: Run eval
        run: |
          grounding eval \
            --agent mini \
            --agents-dir tests/eval_fixtures/agents \
            --fixtures tests/eval_fixtures/mini_fixtures.yaml \
            --corpus tests/eval_fixtures/mini_corpus \
            --embeddings tests/eval_fixtures/mini_index \
            --baseline docs/eval/baselines/mini-corpus.json \
            --fail-under 0.05 \
            --out eval-output/
      - name: Comment on regression
        if: failure()
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const path = require('path');
            const dir = 'eval-output';
            const md = fs.readdirSync(dir).filter(f => f.endsWith('.md'))[0];
            const body = fs.readFileSync(path.join(dir, md), 'utf8');
            // upsert comment with marker <!-- grounding-eval-comment -->
            // ... (use peter-evans/create-or-update-comment in practice)
```

CI has no venv; commands run against the runner's Python directly. Do not prefix with `./venv/bin/`.

### Refresh Procedure (documented in README)

A baseline refresh is intentional, human-reviewed, and PR-gated:

1. Run the eval locally with the new code.
2. Inspect the deltas. Confirm the change is *intended* (e.g., a reranker shipped).
3. Regenerate the baseline JSON.
4. Open a PR titled `eval: refresh mini-corpus baseline (<reason>)`. The PR description must state what changed and why.
5. Reviewer confirms the deltas match the change rationale before approving.

This is documented to prevent baseline-rubber-stamping on regressions.

### Out of Scope

- Per-PR retrieval diff visualization (which queries changed rank).
- Multiple baseline files for different code paths.
- Scheduled (nightly) eval runs against larger corpora.
- LLM-judged generation quality (entirely separate epic).

## Testing

CI changes are validated by behavior, not unit tests. Required validation:

1. **Green path:** Open a no-op doc-only PR touching `docs/eval/README.md`. Confirm the workflow does NOT trigger (paths filter excludes `.md` outside the trigger set) OR triggers and passes cleanly. Document the chosen behavior.
2. **Red path (Task 5):** Deliberate regression PR. Confirm: workflow fails, PR comment posts the Markdown report, comment marker is present, re-run updates the same comment.
3. **Cache effectiveness:** Second run of the same workflow on the same PR completes in under 60s (cache hit on pip + model).
4. **Local reproducibility:** `grounding eval --baseline docs/eval/baselines/mini-corpus.json` run locally produces identical numbers to CI for the same commit.

### Smoke checks before opening the demo PR
- `act` (or equivalent) runs the workflow locally if available; otherwise push to a fork.
- Verify `actions/github-script` permissions (`pull-requests: write`) are granted in the workflow `permissions:` block.

## Risk and Mitigation

| Risk | Severity | Mitigation |
|------|----------|------------|
| CI runtime balloons as fixtures grow | Medium | Mini corpus is intentionally small; cap at ~10 fixtures. Real-agent eval stays local. |
| Model download fails in CI | Medium | Cache the sentence-transformers model dir; pin model name. |
| Baseline drift via rubber-stamped refresh PRs | High | README documents human-review requirement; refresh PR title convention enforced by reviewer norm, not automation (low cost). |
| Workflow comment spam on multiple pushes | Low | Marker-based upsert ensures single comment per PR. |
| `paths:` filter misses a relevant change (e.g., a new module that affects retrieval) | Medium | README documents the filter list as load-bearing; reviewers add the new path when introducing retrieval-affecting code. |

2. **TEST-001 (medium) — AC7 demo regression PR deferred.** Task 5 is explicitly `[ ]` incomplete and noted as "left to the user — requires push access and a disposable branch."
