## Dev Notes

### The mini-corpus saturation problem (carry-over from 18.4)

Story 17.4's baseline shows `recall@1 = recall@5 = MRR = 1.000` and `n_items=3` on the mini corpus. Hybrid retrieval cannot lift a score that's already maxed — same constraint that made 18.4's mini measurement a null result. This is a *feature* of the mini corpus (smoke test for the pipeline), not a bug. AC2 accepts this and requires the real measurement to come from a substantive corpus. If private measurement is not available, the story still ships — with hybrid staying opt-in and the README documenting that the lift is not measurable on public-repo data alone. This is the honest null result and a legitimate outcome, blessed by 18.4's precedent.

### Why flip thresholds are identical to 18.4

`recall@5 lift ≥ 0.03` and `citation_accuracy` non-decreasing — same numbers as 18.4. Rationale holds: published BM25+dense hybrid papers typically report 5–15% relative `recall@5` lifts; on a corpus with `recall@5 ≈ 0.7`, that's +0.035 to +0.105 absolute. 0.03 catches the low end and leaves noise room. `citation_accuracy` is the hard gate: hybrid should not demote citation-relevant chunks, and BM25's lexical bias against paraphrased-but-correct chunks is exactly the failure mode this guard catches.

### Why measure the full four-cell matrix

Measuring only `{hybrid off vs on}` with rerank off tells us whether hybrid helps alone. Measuring only `{rerank off vs on}` with hybrid off tells us whether rerank helps alone (18.4 already did this). The full four-cell matrix additionally tells us:

- **Is hybrid's contribution additive to rerank's?** If `{HH}` > `{Hh}` > `{hh}` and `{HH}` > `{hH}` > `{hh}`, both channels add independently. Expected.
- **Does rerank absorb hybrid's gains?** If `{hH}` ≈ `{HH}`, rerank alone captures most of what hybrid would have contributed (rerank is good enough at promoting the right chunks that BM25's additional signal is redundant). Possible.
- **Does hybrid hurt when combined with rerank?** If `{HH}` < `{hH}`, hybrid is introducing noise that rerank then has to wade through. Unlikely but possible; BM25's lexical overfitting on rare-but-irrelevant tokens is the failure mode.

The epic's decision rule only flips hybrid based on `{hh}` vs `{Hh}` (rerank-held-fixed comparison). The other cells inform the interpretation paragraph and the compound-cost callout in CLAUDE.md, not the flip decision itself.

### Orthogonal flip decisions

Hybrid and rerank flips are independent. If 18.4 produced NO FLIP (it did) and 19.4 produces FLIP, the default becomes `{hybrid on, rerank off}`. If 19.4 also produces NO FLIP, the default stays `{hybrid off, rerank off}`. Users who want both can enable both via flags or config — 19.3 made the composition work. The README comparison matrix shows them what they're buying in each cell.

### Compound latency callout (CLAUDE.md)

Based on 18.4's measured numbers (rerank-base pool=50 ≈ 1.8s on Apple Silicon), the worst case is `{HH}`: BM25 search (milliseconds) + FAISS search (~28ms) + RRF merge (sub-millisecond) + rerank (~1.8s) ≈ ~1.85s per query. The hybrid-only cost is negligible; the rerank cost dominates. Document this explicitly so users don't assume hybrid itself is slow.

### nDCG@10 quirk (TD-002, still open)

Same caveat as 18.4: `ndcg_at_10 > 1.0` due to missing ideal-DCG normalization. Footnote the column in the comparison matrix; `recall@5` and `citation_accuracy` do the load-bearing work for the flip rule, so the nDCG quirk doesn't affect the decision.
