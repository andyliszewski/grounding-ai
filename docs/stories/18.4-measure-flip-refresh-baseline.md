## Story

**As a** grounding-ai maintainer,
**I want** to run the eval harness with reranking on and off, commit the measured comparison, and flip the default only when the lift is real and citation accuracy is non-decreasing,
**so that** Epic 18 closes with published numbers rather than marketing, and downstream users get a default that's been proven rather than asserted.

## Acceptance Criteria

1. A comparison run is executed against the mini corpus with (a) rerank off (baseline), (b) rerank on with `bge-reranker-base`, pool=50. Both runs use identical fixtures and a pinned commit SHA.
2. Because the mini corpus is intentionally small (3-5 chunks, existing recall already near-saturated), the **primary measurement** is run against a more substantive corpus — the maintainer's private `grounding-ai-private` corpus. Private-run numbers are transcribed into the public eval README as an aggregate row, without exposing private corpus content.
3. `docs/eval/README.md` gains a "Reranking comparison" sub-section showing, for each corpus tier:
   - `recall@1`, `recall@5`, `recall@10`, `MRR`, `nDCG@10`, `citation_accuracy` — rerank off vs rerank on.
   - Model, pool size, corpus identifier, commit SHA, UTC timestamp.
   - A one-sentence interpretation per row.
4. Flip-the-default decision rule (documented in the README):
   - Flip to `enabled: true` in `config.example.yaml` **only if** at least one corpus tier shows `recall@5` lift ≥ 0.03 AND `citation_accuracy` non-decreasing vs off-baseline.
   - If neither corpus shows that lift, keep the default `enabled: false` and document why. Epic 18 still closes with plumbing shipped and measured.
5. If the default is flipped: `docs/eval/baselines/mini-corpus.json` is regenerated with rerank-on numbers and `format_version: 2` preserved. PR description includes before/after aggregates per Epic 16 refresh discipline.
6. If the default is not flipped: the mini-corpus baseline stays untouched; rerank-off remains the CI-gated floor.
7. CI workflow (`.github/workflows/eval.yml`) continues to pass on the merge commit. If the default flips, the CI-invoked `grounding eval` command picks up the new default via config without flag changes (Story 18.3's config resolution handles this).
8. `docs/ROADMAP.md` is updated in the merge PR:
   - Tier 1 #1 (cross-encoder reranking) moves to the "Shipped" section with a link to this epic and to the comparison table.
   - Tier 1 #2 (hybrid retrieval) is renumbered to #1 and explicitly marked "Unblocked by Epic 18."
9. `CLAUDE.md` is updated to reflect whichever default the flip decision picked. Latency table added by Story 18.3 is cross-checked against the actual measurements from this story and corrected if off by more than ~30%.
10. A demonstration regression artifact is linked (reuse the Epic 16 / 17 precedent): a deliberate regression PR proving the CI gate fires on a dropped rerank-on number (only if default flipped; otherwise N/A with rationale).
11. Epic 18 is marked "Shipped" in its own file header and in ROADMAP.

## Dev Notes

### The mini-corpus saturation problem

Story 17.4's baseline already shows `recall@1 = recall@5 = MRR = 1.000` and `n_items=3` on the mini corpus. Reranking cannot lift a score that's already maxed. This is a *feature* of the mini corpus (it's a smoke test for the pipeline, not a quality benchmark), not a bug. AC#2 accepts this and requires the real measurement to come from a substantive corpus.

If private measurement is not available, the story should still ship — with rerank staying opt-in and the README documenting that the lift is not measurable on the public-repo data alone. This is the honest null result and a legitimate outcome.

### Why flip thresholds are modest

`recall@5 lift ≥ 0.03` is chosen because:
- Published reranker papers typically report 10-20% relative lifts; on a corpus with `recall@5 ≈ 0.7`, that's +0.07 to +0.14 absolute.
- A 0.03 absolute threshold catches half that effect and still leaves room for noise.
- `citation_accuracy` non-decreasing is a hard gate, not a soft one: reranking *should not* move citation-relevant chunks out of the top results.

### nDCG@10 quirk (known tech debt)

17.4's dev notes flagged that `ndcg_at_10` can exceed 1.0 due to missing ideal-DCG normalization when multiple relevant chunks map to the same doc. This affects the nDCG@10 column in the comparison table. Options:
1. Report nDCG@10 as-is with a footnote explaining the quirk (preferred for this story — keeps scope tight).
2. Fix the normalization first (blocks Epic 18 close; reject for scope reasons).

Recommend option 1. Add a TD entry in `docs/TECH-DEBT.md` pointing to the follow-up.

### Measurement host footnote template

> Latency measurements in the table above were taken on `<CPU model>`, `<N>` cores, `<GB>` GB RAM, Python 3.13.`<x>`, sentence-transformers `<version>`. Numbers on other hardware may vary up to ~30%; the relative ordering (no-rerank < base < large) is stable.

### Out of scope

- Fine-tuning the reranker on the project's own fixture sets.
- Per-agent reranker overrides.
- GPU measurement path.
- nDCG@10 normalization fix (separate TD entry).
- Query rewriting or hybrid retrieval — Tier 1 #2, next epic.
