## Story

**As a** grounding-ai contributor,
**I want** a pure `search_hybrid(query, index_dir, ...)` function that fuses FAISS dense and BM25 lexical rankings via Reciprocal Rank Fusion,
**so that** Story 19.3 can drop it into the three retrieval surfaces without any further design decisions, and 19.4 can measure the lift cleanly.

---

## Acceptance Criteria

1. A new module `grounding/hybrid.py` exports `search_hybrid(query, index_dir, *, top_k, pool_size, k_rrf=60, load_index_fn=None, load_bm25_fn=None, embed_fn=None) -> list[dict]`.
2. Each returned dict carries: `chunk_id`, `doc_id`, `faiss_rank: int | None`, `bm25_rank: int | None`, `rrf_score: float`, `faiss_distance: float | None`, `rank: int` (1-indexed post-fusion rank). No body text, no front matter fields — enrichment is the caller's job (consistent with existing `search_similar_chunks` contract at `vector_store.py:274`).
3. Fusion math: `rrf_score(d) = Σ 1/(k_rrf + r_channel(d))` over channels in which `d` appears. Documents missing from a channel contribute 0 for that channel. Sorting is descending by `rrf_score`, tie-break by `chunk_id` lexicographic for determinism.
4. Pool-size contract: the function fetches `pool_size` candidates from each channel (so up to `2 * pool_size` unique chunks enter the merge), sorts by `rrf_score`, and returns the top `top_k`.
5. Missing BM25 artifacts (agent embedded before 19.1 shipped, or BM25 build failed) → function falls back to dense-only: calls FAISS, assigns `bm25_rank=None` to every result, logs `WARNING grounding.hybrid: BM25 artifacts missing at <dir>; running dense-only`, returns with `rrf_score` equal to `1/(k_rrf + faiss_rank)` (so downstream reranking / ordering still works). Every returned dict carries `hybrid_degraded: True` in this path; absent otherwise.
6. Asymmetric-tombstone defense: if a `chunk_id` is tombstoned in **either** channel's map, it is dropped from the merge regardless of whether the other channel surfaced it. A DEBUG-level log records the first such drop per call (not per candidate).
7. Empty query tokens (after `tokenize(query)`) → BM25 channel contributes nothing; dense channel runs normally; `hybrid_degraded` is **not** set (this is normal behavior for single-word stopword queries).
8. Empty result from both channels → returns `[]`; no error.
9. `search_hybrid` is a pure function of its inputs (plus the files at `index_dir`); injected function kwargs (`load_index_fn`, `load_bm25_fn`, `embed_fn`) allow fully-stubbed unit tests without real FAISS / BM25 / embedder instances.
10. Unit tests in `tests/test_hybrid.py` cover: RRF math against the documented worked example (AC#3), dense-only fallback when BM25 absent, both-channels-empty, pool-size truncation, tombstone cross-channel filtering, asymmetric tombstones, empty-query-tokens, tie-break determinism, `hybrid_degraded` flag.
11. No new runtime dependencies.
12. No changes to `grounding/vector_store.py`, `grounding/bm25.py`, `grounding/eval/runner.py`, or any retrieval surface. Hybrid is purely additive.

## Dev Notes

### Worked example for RRF (committed to the test suite)

Inputs (1-indexed ranks):
- FAISS: `[A, B, C, D]` → ranks 1, 2, 3, 4.
- BM25: `[B, E, A]` → ranks 1, 2, 3.
- `k_rrf = 60`.

Calculations:
- `RRF(A) = 1/(60+1) + 1/(60+3) = 0.016393 + 0.015873 = 0.032266`
- `RRF(B) = 1/(60+2) + 1/(60+1) = 0.016129 + 0.016393 = 0.032522`
- `RRF(C) = 1/(60+3)               = 0.015873`
- `RRF(D) = 1/(60+4)               = 0.015625`
- `RRF(E) = 1/(60+2)               = 0.016129`

Final order: `B, A, E, C, D`. Test asserts exact list equality plus approximate `rrf_score` equality.

### Why ids-only return shape (no body, no front matter)

`search_similar_chunks` (`vector_store.py:274`) returns `list[tuple[chunk_id, distance]]` and lets callers enrich with body + front matter — e.g. `SearchCorpusTool._read_chunk:212`. Matching that contract keeps hybrid pure and avoids coupling the module to corpus-directory layout. Story 19.3 builds the same enrichment layer once in the retrieval surfaces and both channels feed through it. Epic 19's AC#2 ("returns dicts matching the existing retrieval-result shape") is preserved at the *dict-shape* level (keys present are a subset of the existing shape's keys); body/source filling is downstream.

### Tombstone cross-check contract

Each channel already filters its own tombstones at search time (Story 14.1 for FAISS, Story 19.1 for BM25). The cross-check here defends against partial-rebuild inconsistency: rare, but a real failure mode when the watcher updates FAISS and BM25 non-atomically. Filtering on the union of both tombstone sets is cheap and eliminates the phantom-candidate class of bugs.

### Injected function kwargs

```python
def search_hybrid(
    query: str,
    index_dir: Path,
    *,
    top_k: int,
    pool_size: int,
    k_rrf: int = 60,
    load_index_fn=None,    # defaults to vector_store.load_vector_index
    load_bm25_fn=None,     # defaults to grounding.bm25.load_bm25_index
    embed_fn=None,         # defaults to grounding.embedder.embed_query
) -> list[dict]:
```

Matches the pattern Story 16.2 established for `run_eval`. Production code calls with defaults; tests inject stubs.

### Interaction with reranker (for 19.3 awareness)

Story 19.3 will call `search_hybrid(...)` to produce the candidate pool, then optionally pass the pool through `grounding.reranker.rerank(...)`. The reranker's contract (18.1 AC #4) preserves input dict keys and adds `rerank_score` / `faiss_distance`. Since 19.2 returns dicts with `rrf_score` and `bm25_rank` as separate keys, rerank will flow through without key collision: `score` is empty on 19.2 output (bodies get enriched and `score` populated in 19.3 before rerank). This is fine; document explicitly in 19.3.

### Performance note

With `pool_size=50`: up to 100 unique candidates enter the merge, dict accumulation is O(200) operations, sort is O(N log N) on ≤100 items. Sub-millisecond. BM25 search on `rank_bm25` is O(vocab) — typically milliseconds. Dense search is the dominant cost. No optimization needed.
