## Story

**As a** corpus administrator,
**I want** to append new embeddings to an existing FAISS index without full regeneration,
**so that** embedding updates complete in minutes rather than hours for large corpora.

---

## Acceptance Criteria

1. New function `append_to_vector_index()` adds embeddings to existing FAISS index
2. Chunk map updated atomically with new entries appended
3. Tombstone mechanism marks deleted documents without requiring index rebuild
4. Document updates detected and handled (old chunks tombstoned, new chunks appended)
5. Staleness check function returns list of new/deleted/updated documents
6. Unit tests validate append, tombstone, and update operations
7. No regression to existing `write_vector_index()` functionality

### FAISS Incremental API

**Append Pattern**:
```python
# Load existing index
index = faiss.read_index(str(index_path))

# Prepare new embeddings as float32 matrix
new_matrix = np.array(new_embeddings, dtype=np.float32)

# Append to index (vectors added at positions index.ntotal onwards)
start_idx = index.ntotal
index.add(new_matrix)

# Save updated index
faiss.write_index(index, str(temp_path))
temp_path.replace(index_path)
```

**Key Constraint**: FAISS IndexFlatL2 does NOT support:
- Removing vectors by index
- Updating vectors in place

This is why we use tombstones in the chunk map rather than modifying the FAISS index.

[Source: FAISS documentation, TD-001 Technical Analysis]

### Tombstone Strategy

**Why Tombstones**:
1. FAISS IndexFlatL2 doesn't support deletion
2. Rebuilding 167k vectors takes ~3.5 hours
3. Deletions are rare compared to additions
4. Tombstone overhead is minimal (one JSON field)

**Query-Time Filtering**:
```python
def search_similar_chunks(index, chunk_map, query_embedding, top_k):
    # Request extra results to account for tombstones
    tombstone_ratio = chunk_map.get("tombstone_count", 0) / chunk_map.get("index_size", 1)
    fetch_k = int(top_k * (1 + tombstone_ratio + 0.1))  # 10% buffer

    distances, indices = index.search(query, fetch_k)

    # Filter tombstoned chunks
    results = []
    for idx, distance in zip(indices[0], distances[0]):
        chunk = chunks[idx]
        if chunk.get("deleted_utc") is None:
            results.append((chunk["chunk_id"], distance))
            if len(results) >= top_k:
                break

    return results
```

**Rebuild Heuristic**:
- Tombstone ratio > 30% → recommend rebuild
- Logged as warning during search operations
- CLI `--check` flag reports rebuild recommendation

### Staleness Detection

**StalenessReport Dataclass**:
```python
@dataclass
class StalenessReport:
    new_docs: List[str]      # doc_ids in manifest but not in index
    deleted_docs: List[str]  # doc_ids in index but not in manifest
    updated_docs: List[str]  # doc_ids with changed file hashes
    is_stale: bool           # True if any of the above are non-empty
    tombstone_ratio: float   # Current tombstone percentage
    should_rebuild: bool     # True if tombstone_ratio > threshold
```

**Detection Logic**:
1. Load corpus manifest (`_index.json`)
2. Load chunk map and extract active doc_ids
3. Compare sets to find new/deleted
4. For potential updates, compare file hashes in manifest vs chunk map

[Source: TD-001 Technical Analysis]

### Concurrency Considerations

**Risk**: Watcher may trigger embedding update while another is in progress.

**Mitigation** (implemented in Story 14.3):
- Lock file mechanism (`_embeddings.lock`)
- This story focuses on atomic writes only
- Concurrent reads are safe (FAISS supports this)

[Source: TD-001 Edge Cases]

### Error Handling Strategy

**Scenarios**:
1. **Index not found**: Fall back to `write_vector_index()` (create new)
2. **Corrupted chunk map**: Raise clear error, don't attempt recovery
3. **Dimension mismatch**: Validate before append, raise ValueError
4. **Disk full during write**: Atomic write pattern ensures no corruption
5. **Empty new_embeddings**: Log info, return early (no-op)
