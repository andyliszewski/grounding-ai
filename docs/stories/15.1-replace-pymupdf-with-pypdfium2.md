## Dev Notes

### API Migration Reference

pypdfium2 provides equivalent PDF-to-image rendering via PDFium (the Chromium PDF engine). The API mapping is straightforward:

| PyMuPDF (fitz) | pypdfium2 | Notes |
|----------------|-----------|-------|
| `fitz.open(path)` | `pdfium.PdfDocument(path)` | Constructor-based opening |
| `page.get_pixmap(matrix=fitz.Matrix(2, 2))` | `page.render(scale=2)` | Returns `PdfBitmap` |
| `page.get_pixmap(dpi=150)` | `page.render(scale=150/72)` | DPI to scale: `dpi/72` |
| `Image.frombytes("RGB", [w, h], samples)` | `bitmap.to_pil()` | Direct PIL conversion |
| `len(doc)` | `len(doc)` | Same interface |
| `doc.close()` | `doc.close()` | Same interface |
| `doc[page_num]` | `doc[page_num]` | Same page indexing |

### Licensing Context

- **PyMuPDF**: AGPL-3.0 (copyleft, requires derivative works to also be AGPL)
- **pypdfium2**: Apache-2.0 / BSD-3-Clause (permissive, compatible with MIT)
- **grounding-ai**: MIT license
- AGPL-3.0 dependency in an MIT project creates a licensing conflict that prevents clean open-source release

### pypdfium2 render() Return Type

`page.render()` returns a `PdfBitmap` object. Key method:
- `bitmap.to_pil()` returns a `PIL.Image.Image` in RGB mode

This replaces the three-step PyMuPDF pattern (`get_pixmap` -> extract dimensions/samples -> `Image.frombytes`) with a single call.

### Import Pattern Difference

PyMuPDF uses the alias `fitz` which is the legacy module name. pypdfium2 uses the conventional `import pypdfium2 as pdfium` pattern.

For `hybrid_processor.py`, maintain the try/except ImportError guard since pypdfium2 is an optional dependency for the hybrid processing path:

```python
try:
    import pypdfium2 as pdfium
except ImportError:
    pdfium = None
```

For `omr_parser.py`, maintain the lazy import inside `detect_music_content()` since it's also conditionally required:

```python
try:
    from PIL import Image
    import pypdfium2 as pdfium
except ImportError:
    logger.warning("Pillow or pypdfium2 not available for staff line detection.")
    return False
```

For `formula_extractor.py`, the top-level import is appropriate since the module requires the dependency unconditionally.

### Testing Strategy

The mock structure changes because pypdfium2 has a different object hierarchy:
- Instead of mocking a pixmap with `.width`, `.height`, `.samples`, mock the `render()` return to have a `.to_pil()` method that returns a PIL Image
- Instead of patching `fitz.open`, patch `pdfium.PdfDocument`
- The page indexing (`doc[n]`) and document length (`len(doc)`) interfaces are the same
