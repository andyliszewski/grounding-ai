## Story Description

As a **developer**, I want to **connect all pipeline stages into a single controller** so that the CLI can process a folder end-to-end and produce the corpus outputs (doc/chunks/meta/manifest).

This story wires scanner → parser → formatter → chunker → metadata generator → writer → manifest manager, ensuring `FileContext` carries state across each step and the CLI command executes the full run.

---

## Context

### From PRD v0.2  
[Source: docs/prd.md — Sections "Batch Processing" & "Output Guarantees"]

- CLI command should walk the entire PDF directory and produce complete corpus output deterministically.  
- Processing must continue even if individual files fail (handled in Story 5.2).  
- Outputs must align with architecture contract.

### From Architecture v0.2  
[Source: docs/architecture.md — Sections 4 Pipeline Overview & 5 Data Contracts]

- Pipeline executes sequential stages using shared `FileContext`.  
- Chunk metadata and hashes generated in Epic 3 feed directly into writer.  
- Manifest updated per processed document.

### From Epic 5
- Integrate all previously implemented modules.  
- Ensure CLI uses pipeline controller rather than direct stage calls.  
- Provide single entry point for subsequent QA testing.

---

## Acceptance Criteria

### AC1: Controller Implementation
- ✅ Introduce `pdf2llm.controller.run(config: PipelineConfig, files: Sequence[Path] | None = None) -> PipelineResult` that invokes the full stage chain.  
- ✅ Accept optional pre-scanned list for tests; otherwise call `scan_pdfs`.  
- ✅ Returns `PipelineResult` with contextual data for summaries.

### AC2: CLI Integration
- ✅ Update `pdf2llm.cli` to call `controller.run` instead of `run_pipeline` directly.  
- ✅ Ensure CLI still supports dry-run (controller must respect `dry_run`).  
- ✅ CLI exit codes/summary remain intact.

### AC3: FileContext Propagation
- ✅ Each stage receives the same `FileContext` instance, updated with parser/formatter/chunker outputs.  
- ✅ Chunker output stored for metadata/writer use.  
- ✅ Metadata builder invoked with actual params/tooling information from pipeline config.

---

## Implementation Guidance

1. **Controller Module:** Add `pdf2llm/controller.py` coordinating stage calls; reuse pipeline helpers.  
2. **Chunk Generation:** Call `split_markdown` (Story 3.1) and feed results into `build_chunk_metadata`/writer.  
3. **Metadata:** Use `build_meta_yaml` to generate meta files before writer call.  
4. **Manifest:** Register document entry after successful write (or skip when dry-run).  
5. **Tests:** Reuse fixtures from `tests/test_pipeline.py` to create integration-style tests under `tests/test_integration.py`.
