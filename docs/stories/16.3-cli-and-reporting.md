## Story

**As a** grounding-ai contributor or maintainer,
**I want** a `grounding eval` CLI command that runs the harness, writes Markdown + JSON reports, and can compare against a committed baseline,
**so that** retrieval changes can be measured locally and a regression-gate is available for CI to plug into.

---

## Acceptance Criteria

1. `grounding eval` subcommand registered in the existing argparse CLI (follow precedent: `_create_agents_parser`, `_create_embeddings_parser` in `grounding/cli.py`); `grounding eval --help` lists all flags.
2. Flags: `--agent <name>` (required), `--agents-dir <path>` (default: `agents/`), `--fixtures <path>` (default: `docs/eval/fixtures/<agent>.yaml`), `--corpus <path>` (default from existing config resolution), `--embeddings <path>` (default: `embeddings/<agent>/`), `--top-k <int>` (default: 10), `--out <dir>` (default: `docs/eval/reports/`), `--baseline <path>` (optional), `--fail-under <float>` (optional, default: 0.0).
3. A successful run writes two artifacts to `--out`: `<run-id>.md` (human-readable) and `<run-id>.json` (machine-readable). `<run-id>` format: `<agent>-<UTC-yyyymmdd-hhmmss>`.
4. Markdown report sections: header (agent, fixture path, run timestamp, corpus snapshot info), aggregate metrics table, per-tag breakdown table, per-item table (item_id, query, first_hit_rank, top-3 retrieved doc_ids), skipped items list.
5. JSON artifact serializes the full `EvalRun` dataclass tree (stable schema; `format_version: 1`).
6. With `--baseline <path>`, the CLI prints a delta table (current vs baseline for each aggregate metric and per-tag entry) to stdout and includes it in the Markdown report.
7. `--fail-under <float>` exits with code 1 if **any** aggregate metric drops by more than the given absolute delta vs baseline (e.g., `--fail-under 0.02` means a 0.02 absolute drop in `recall@5` triggers failure). Without `--baseline`, `--fail-under` is ignored with a warning.
8. Stdout summary (always printed) is concise: agent, n_items, n_skipped, recall@1/3/5, MRR, plus baseline deltas if applicable. Designed to fit in a CI log preview.
9. Unit tests cover: report serialization (JSON round-trip), Markdown generation against a synthetic `EvalRun`, baseline diff computation, exit-code policy.
10. Smoke test invokes `grounding eval --help` and the actual command against the mini corpus from Story 16.2.

## Dev Notes

### CLI Defaults Resolution

Follow precedent from `grounding agents` and `grounding embeddings` subcommands. Default paths should resolve in this order:

1. Explicit flag value.
2. `config.yaml` if present (the repo currently ships `config.example.yaml` only; treat config-driven defaults as opt-in).
3. Conventional location (e.g., `./embeddings/<agent>/`, `./docs/eval/fixtures/<agent>.yaml`).

If none resolves to an existing path, exit with a clear "could not find X; pass `--<flag>`" message and exit code 2. Do not silently fall back to creating directories.

### JSON Schema (canonical)

```json
{
  "format_version": 1,
  "run_id": "data-scientist-20260413-153000",
  "agent": "data-scientist",
  "fixture_path": "docs/eval/fixtures/data-scientist.yaml",
  "corpus_dir": "/path/to/corpus",
  "embeddings_dir": "/path/to/embeddings/data-scientist",
  "top_k": 10,
  "started_utc": "2026-04-13T15:30:00Z",
  "finished_utc": "2026-04-13T15:30:42Z",
  "aggregate": {
    "n_items": 12,
    "n_skipped": 1,
    "recall_at_1": 0.50,
    "recall_at_3": 0.75,
    "recall_at_5": 0.83,
    "recall_at_10": 0.92,
    "mrr": 0.61,
    "ndcg_at_10": 0.72,
    "per_tag": {
      "methodology": {"recall_at_5": 0.80, "mrr": 0.62, "n_items": 5, "low_sample": false}
    }
  },
  "items": [ /* EvalItemResult dicts */ ],
  "skipped": ["ds-007"]
}
```

**Fixture:** docs/eval/fixtures/data-scientist.yaml (12 items, 1 skipped)
**Corpus:** /path/to/corpus

**Duration:** 42s

|--------|-------|-------------|
| recall@1 | 0.50 | ↓0.02 |
| recall@5 | 0.83 | ↑0.04 |
| MRR | 0.61 | = |

| Tag | n | recall@5 | MRR | low_sample |
|-----|---|----------|-----|------------|
| methodology | 5 | 0.80 | 0.62 | no |

| ID | Query | First Hit | Top-3 doc_ids |

### Baseline File Format

A baseline file is just the `aggregate` subtree of an `EvalRun` JSON, plus minimal metadata (agent, captured_utc, fixture_version). Story 16.4 will commit one and document the refresh procedure.

```json
{
  "format_version": 1,
  "agent": "data-scientist",
  "captured_utc": "2026-04-13T15:30:00Z",
  "fixture_version": 1,
  "aggregate": { ... }
}
```

### Exit Code Policy

| Scenario | Exit |
|----------|------|
| Successful run, no `--baseline` | 0 |
| Successful run with `--baseline`, all metrics within `--fail-under` | 0 |
| Run with `--baseline`, any metric drop > `--fail-under` | 1 |
| Fixture file not found / agent unknown | 2 |
| Embeddings index missing | 3 |
| Unexpected exception | 4 |

Use distinct codes so CI can branch logic if needed.

### `worst_drop` Definition

Across all aggregate metrics (excluding `n_items`, `n_skipped`, and `low_sample` per-tag groups), compute `baseline_value - current_value` and take the maximum. Negative values (improvements) are clamped to 0. This is the value compared against `--fail-under`.

### Stdout Summary (target)

```
eval: data-scientist  items=12  skipped=1
  recall@1 0.50 ↓0.02   recall@3 0.71 =       recall@5 0.83 ↑0.04
  recall@10 0.92 ↑0.01  MRR 0.61 =            nDCG@10 0.72 ↑0.03
  worst_drop 0.02  fail_under 0.05  → PASS
```
