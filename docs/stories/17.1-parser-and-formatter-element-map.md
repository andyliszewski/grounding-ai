# Story 17.1: Parser & Formatter Element Map

**Epic:** Epic 17 - Page and Section Citations
**Status:** Done
**Owner:** Product Owner (Sarah)
**Created:** 2026-04-14
**Dependencies:** none (first story in Epic 17)
**Related:** ROADMAP Tier 1 #2

---

## Story

**As a** grounding-ai contributor,
**I want** the Markdown formatter to emit a parallel element map recording which character ranges belong to which source page and heading context,
**so that** the chunker (Story 17.2) can compute accurate `page_start`, `page_end`, and `section_heading` for every chunk.

---

## Acceptance Criteria

1. A new public API `format_markdown_with_map(elements, ...) -> FormatResult` lives in `grounding/formatter.py` and returns both the formatted Markdown string and an ordered list of `FormattedElement` records.
2. Existing `format_markdown(...)` continues to work unchanged (returns just the string). Existing callers are not modified in this story.
3. `FormattedElement` is a frozen dataclass with fields: `char_start: int`, `char_end: int`, `page_number: Optional[int]`, `heading_stack: tuple[str, ...]`.
4. For Unstructured elements: `page_number` comes from `element.metadata.page_number` when present; `heading_stack` is derived from preceding elements whose category is `Title` (or equivalent heading type).
5. For `TextElement` (pdftotext fallback): `page_number` is `None` (fallback path has no page info); `heading_stack` is empty. Document this explicitly.
6. For Marker-produced Markdown (already-rendered string input, not element sequence): `page_number` is populated if the Marker per-block page info is threaded through; if not available in this story's scope, `page_number=None` and the limitation is documented in Dev Notes for Story 17.2 to address or defer.
7. Element map is ordered by `char_start` and spans are non-overlapping; the union covers every non-whitespace region of the formatted Markdown. Inter-block whitespace (blank lines between paragraphs) may be unmapped.
8. Heading stack correctly nests: `h1 → (h1,)`, then `h2 under h1 → (h1, h2)`, then new `h1 → (h1',)` (pops back up).
9. Unit tests in `tests/test_formatter_element_map.py` cover: multi-page PDF with heading hierarchy, single-page PDF no headings, EPUB (no pages, has headings), raw Markdown input (no pages, has headings via `#`/`##`), pdftotext fallback (no pages, no headings).
10. No new runtime dependencies.

### Completion Notes List
- Added `FormattedElement` and `FormatResult` frozen dataclasses plus a public `format_markdown_with_map(...)` entry point to `grounding/formatter.py`. Existing `format_markdown` left bit-for-bit identical; no callers modified (Story 17.2 will wire pipeline/controller).
- Heading detection dispatch is per-element, not per-input-batch: Unstructured elements use `category in {"Title", "Header"}` with level derived from `metadata.category_depth` (0-indexed → 1-indexed; default 1); `TextElement` inputs detect a leading `^#{1,6}\s+...` line as a heading so the same path serves EPUB ebooklib fallback and raw-Markdown ingest. pdftotext `TextElement`s naturally stay `heading_stack=()` because they contain no `#` lines.
- `_compute_heading_stacks` is the single source of truth for nesting — headings pop all entries at their level or deeper before pushing themselves, matching AC #8 exactly.
- Position math: each kept element's normalized text is joined with a fixed `\n\n` separator after a per-element `_normalize_block_text` (line-ending normalization + per-line rstrip + trailing-newline strip). This makes `char_start`/`char_end` a trivially-derivable running offset and guarantees non-overlapping, ordered spans. Front matter, when present, shifts all offsets by `len(front_matter)`.
- `_coverage_check` verifies ordering, non-overlap, and that every non-whitespace char in `[first.char_start, last.char_end)` is covered by some span — asserted on every test scenario per AC #7 and #9.
- Marker-produced Markdown is handled via the TextElement/raw-Markdown path (no pages, headings from `#`/`##`) per Dev Notes guidance; deferred to 17.2 for sidecar-aware wiring.
- No new runtime dependencies (AC #10). Only stdlib additions: `re`, `dataclasses.dataclass`, extra typing imports.

### Unstructured element categories treated as headings

Start with the `Title` category only. Record this in the module docstring. If spot-checks on real PDFs show that other categories (e.g., `Header`) should count, handle in a follow-up — do not expand scope here.

### Marker path — out of scope for this story

Marker produces Markdown strings, not element sequences. Extracting per-block page info from Marker requires reading its JSON sidecar output rather than the markdown. Deferred to Story 17.2's risk section; this story handles Marker-produced Markdown the same way it handles raw Markdown input (headings from `#`/`##` lines, no pages). Document the limitation.

### Determinism

All emitted element records must be a pure function of input. No timestamps, no object ids. The existing `format_markdown` is already deterministic; preserve that.

### Backward compatibility

The new function `format_markdown_with_map` is additive. `format_markdown` is preserved bit-for-bit — including its call sites in `pipeline.py` and `controller.py`. Story 17.2 is the one that switches callers over.

### Reviewed By: Quinn (Test Architect)

### Code Quality Assessment

Clean, focused implementation. The element-map API is purely additive: `format_markdown` is preserved bit-for-bit and its 12 existing tests pass unmodified. Frozen dataclasses (`FormattedElement`, `FormatResult`) give downstream consumers (Story 17.2) a stable contract with structural type-checking. Helpers are module-private with single responsibilities (`_normalize_block_text`, `_compute_heading_stacks`, `_classify_element`, `_unstructured_page_number`, `_unstructured_heading_level`, `_coverage_check`), and the module docstring clearly states heading-detection rules, Marker deferral, and TextElement semantics. Position math uses a running offset rather than per-element string searches, so span construction is O(n) and deterministic.

The `_coverage_check` invariant (ordering + non-overlap + non-whitespace coverage) is exercised on every scenario per AC #7 and #9, which is the right level of rigor for a contract this load-bearing for downstream chunking.
