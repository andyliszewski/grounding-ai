## Dev Notes

### Architecture Context

pdf2llm follows a modular pipeline architecture where each stage is independent and replaceable. The parser module (`pdf2llm/parser.py`) currently supports Unstructured and Marker parsers using a lazy-loading pattern with importlib.

[Source: docs/architecture.md - System Composition, parser.py implementation]

**Current Parser Architecture**:
```python
# Existing pattern in parser.py
_partition_pdf = None

def _get_partition_pdf():
    global _partition_pdf
    if _partition_pdf is None:
        module = importlib.import_module("unstructured.partition.pdf")
        _partition_pdf = module.partition_pdf
    return _partition_pdf
```

**OMR Parser Integration**:
Follow the same lazy-loading pattern for Audiveris to avoid import errors if OMR dependencies not installed:

```python
# New pattern in omr_parser.py
_audiveris_gateway = None

def _init_audiveris_gateway():
    global _audiveris_gateway
    if _audiveris_gateway is None:
        from py4j.java_gateway import JavaGateway
        # Initialize gateway to Audiveris Java process
        _audiveris_gateway = JavaGateway()
    return _audiveris_gateway
```

[Source: pdf2llm/parser.py:31-44 - Lazy loading pattern]

### Audiveris Integration Strategy

Based on Story 7.1 research (Audiveris selected), integration follows these steps:

1. **JRE Detection**: Check for JRE >=11 using subprocess (`java -version`)
2. **Audiveris Location**: Environment variable `AUDIVERIS_HOME` or default paths
3. **Gateway Setup**: py4j connects to Audiveris Java process
4. **Processing**: Audiveris CLI: `audiveris -batch -export <pdf>` → MusicXML output
5. **Parsing**: music21 parses MusicXML → Python objects → MusicElement list

**Alternative Approach** (if py4j proves complex):
- Use subprocess to call Audiveris CLI directly
- Parse MusicXML output file with music21
- Simpler but less performant (process spawn overhead)

[Source: docs/epics/epic-7-omr-support-v02.md - Audiveris Integration]

### MusicElement Data Structure

Represents a single musical element extracted from PDF:

```python
from dataclasses import dataclass
from typing import Optional, Literal

@dataclass
class MusicElement:
    element_type: Literal["note", "rest", "clef", "key_sig", "time_sig", "barline"]
    measure_number: int
    staff_number: int = 1
    voice_number: int = 1

    # Note-specific fields
    pitch: Optional[str] = None  # e.g., "C4", "F#5"
    duration: Optional[float] = None  # in quarter notes (0.5 = eighth note)

    # Metadata
    metadata: dict = None  # tempo, dynamics, etc.

    def __post_init__(self):
        if self.metadata is None:
            self.metadata = {}
```

This structure will be consumed by `music_formatter.py` in Story 7.3.

### Music vs Text Detection Heuristics

**Requirement**: >80% accuracy distinguishing music notation PDFs from text PDFs.

**Detection Approach**:
1. **Quick heuristic** (fast, ~90% accuracy):
   - Count horizontal lines across page (staff lines are evenly spaced)
   - Music notation: 5-line staves, multiple per page
   - Use PIL or OpenCV for line detection

2. **Fallback heuristic** (slower, more accurate):
   - Run Audiveris on first page only
   - If MusicXML output contains <measure> and <note> elements → music
   - Otherwise → text

**Implementation**:
```python
def detect_music_content(file_path: Path) -> bool:
    # Try quick heuristic first
    if _detect_staff_lines(file_path):
        return True

    # Fallback: sample first page with Audiveris
    return _audiveris_sample_page(file_path, page_num=0)
```

This function will be used by `hybrid_processor.py` in Story 7.4.

[Source: Epic 7 Story 7.4 requirements - Hybrid Document Processing]
