## Acceptance Criteria

1. The fixture schema (Story 16.1) gains optional fields on `FixtureItem.expected`: `page` (int or `[start_int, end_int]`) and `section` (str). Schema docs in `docs/eval/README.md` updated with examples and a note that both are optional.
2. `grounding/eval/fixtures.py` loader accepts and validates the new fields. Validation rules: `page` is either a positive int or a 2-element list of positive ints where `start <= end`; `section` is a non-empty string. Bad values produce `FixtureValidationError` with `field="expected.page"` / `field="expected.section"`.
3. `grounding/eval/metrics.py` adds `citation_accuracy(item_results) -> float`: fraction of items whose first-hit retrieved chunk matches the fixture's page/section expectations. Pure function, unit-tested.
4. Hit semantics for citation accuracy:
   - If `expected.page` is an int `N`: retrieved chunk matches when `page_start <= N <= page_end`.
   - If `expected.page` is `[start, end]`: retrieved chunk matches when its `[page_start, page_end]` overlaps the expected range.
   - If `expected.section` is set: retrieved chunk matches when its `section_heading == expected.section` (exact string match; documented as case-sensitive).
   - When both fields are set: both must match.
   - Items with neither `expected.page` nor `expected.section` are excluded from the metric (reported as `n_citation_items` separate from `n_items`).
5. `EvalAggregate` gains two fields: `citation_accuracy: float | None` and `n_citation_items: int`. `None` when no fixture items carry citation expectations.
6. `EvalRun` JSON bumps to `format_version: 2`. Readers handle both versions; `format_version: 1` is parsed as if `citation_accuracy=None` and `n_citation_items=0`. A brief migration note in `grounding/eval/report.py`'s module docstring explains the bump.
7. Runner (`grounding/eval/runner.py`) propagates `page_start`, `page_end`, `section_heading` into each `RetrievedChunk` by reading them from the chunk_map or by loading the chunk's YAML front matter during result assembly. Decision and rationale documented in Dev Notes.
8. Markdown report from Story 16.3 (`grounding/eval/report.py:to_markdown`) surfaces `citation_accuracy` in the aggregate metrics table when non-null and adds a new "Citation" column to the per-item table showing retrieved page/section for each hit.
9. Baseline diff (`grounding/eval/baseline.py`) treats `citation_accuracy` like other aggregate metrics: included in `worst_drop` computation when present in both current and baseline.
10. Mini-corpus fixtures (`tests/eval_fixtures/mini_fixtures.yaml`) are extended with citation expectations on at least 2 items.
11. Mini-corpus chunks are re-ingested from source (post-17.2 pipeline) so `page_start` / `section_heading` are populated. Chunks committed.
12. `docs/eval/baselines/mini-corpus.json` refreshed: new numbers, new `format_version: 2`. Refresh PR follows the Epic 16 refresh discipline (title convention, human-review requirement, rationale in PR description).
13. CI workflow `.github/workflows/eval.yml` stays green on the main merge commit.
14. A demonstration regression PR (new or extending the 16.4 demo) drops `section_heading` from chunks and confirms the gate fires on `citation_accuracy` drop. PR closed without merge; URL recorded in Change Log.
15. `docs/eval/README.md` documents the `citation_accuracy` metric, its semantics, and how to add page/section expectations to fixtures.
16. ROADMAP Tier 1 #2 moved from "next up" to "Shipped" in the same merge PR; Tier 1 #3 (cross-encoder reranking) explicitly called out as unblocked.

## Dev Notes

### Schema example (for README)

```yaml
agent: mini
version: 1
items:
  - id: mini-001
    query: "What does the paper conclude about bootstrap CI coverage?"
    expected:
      doc_ids: ["7a9b2c1f"]
      page: 247                          # single-page expectation
      section: "3.2 Bootstrap Methods"   # exact string match
    tags: ["methodology"]

  - id: mini-002
    query: "How does falsifiability relate to demarcation?"
    expected:
      doc_ids: ["3c4d5e6f"]
      page: [15, 18]                     # range expectation
    tags: ["epistemology"]
```

### Hit-rule worked examples

| Retrieved chunk | Expected | Hit? |
|-----------------|----------|------|
| `page_start=247, page_end=247, section="3.2 ..."` | `page=247, section="3.2 ..."` | ✅ both match |
| `page_start=247, page_end=249, section="3.2 ..."` | `page=248` | ✅ 247 ≤ 248 ≤ 249 |
| `page_start=15, page_end=16, section=null` | `page=[15, 18]` | ✅ ranges overlap |
| `page_start=20, page_end=21` | `page=[15, 18]` | ❌ no overlap |
| `section="3.2 Bootstrap Methods"` | `section="3.2 bootstrap methods"` | ❌ case-sensitive by design |
| `page_start=null, section=null` | `page=247` | ❌ no metadata to match against |

Case-sensitivity is documented; relaxation can be a follow-up if fixture authors find it painful.

### Why `n_citation_items` is separate from `n_items`

An eval set can mix retrieval-only items (legacy, or cases where citation isn't expected) with citation-checked items. Reporting the two counts separately keeps `recall@k` interpretation stable while letting `citation_accuracy` reflect only the subset it applies to. The Markdown report makes both visible so small-N citation sets don't get over-read.

### Runner: reading YAML front matter per retrieved chunk

Each chunk is already on disk at `<corpus>/<slug>/chunks/ch_NNNN.md`. Reading and parsing its YAML front matter on a top-10 result set is ~10 small-file reads per query — negligible. Avoid adding pages/section to `chunk_map` unless profiling shows the per-query reads are meaningful; that can be a follow-up optimization.

### `format_version` bump and backward read

- Writer: always emits `format_version: 2`.
- Reader (`load_baseline`): accepts both `1` and `2`; coerces `1` to v2 in-memory with `citation_accuracy=None, n_citation_items=0`.
- Story 16.3's tests asserting `format_version: 1` must be updated to assert either `1 or 2` where strictness is about readability, or pinned to `2` where the writer is exercised. Dev will audit.

### Chunk-id format nit (from 16.2 follow-up)

Story 16.2 noted the runner tolerates both `<doc_id>/ch_NNNN` (schema doc) and `<doc_id>-NNNN` (actual in-repo format). That tolerance is unchanged by this story; citation accuracy operates on `page_start`/`section_heading`, not chunk_id. No action required here, but worth not forgetting when 17.4 touches the runner's result assembly.

### nDCG quirk (from 16.2 follow-up)

Story 16.2 flagged that `ndcg_at_10` can exceed 1.0 because the implementation doesn't divide by ideal DCG when multiple relevant chunks map to the same doc. Out of scope for 17.4 but good to flag in a fresh tech-debt entry if not already tracked — keeping the baseline stable is more important than the math fix for this epic.

### Out of scope

- Fuzzy section matching (e.g., `"3.2 Bootstrap"` matches `"3.2 Bootstrap Methods"`). Defer.
- Citation accuracy at rank > 1 (e.g., `citation_accuracy@5`). Defer.
- LLM-judged citation correctness. Belongs in a future generation-eval epic.
- nDCG ideal-DCG normalization fix. Separate tech-debt item.
