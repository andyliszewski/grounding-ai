## Story

**As a** grounding-ai user,
**I want** consistent `--rerank` / `--rerank-model` / `--rerank-pool-size` flags on every CLI, a documented `retrieval.rerank` block in `config.yaml.example`, and a richer MCP tool schema,
**so that** I can turn reranking on, choose a model, and tune pool size without editing code â€” from `grounding eval`, `local_rag.py`, and any MCP client alike.

---

## Acceptance Criteria

1. `grounding eval` subcommand exposes four flags: `--rerank` (bool, default false), `--rerank-model` (str, default `BAAI/bge-reranker-base`), `--rerank-pool-size` (int, default 50), `--rerank-top-k` (int, optional â€” overrides `--top-k` for the post-rerank truncation; when omitted, the existing `--top-k` applies to the final output).
2. `scripts/local_rag.py` exposes the same four flags with the same names, defaults, and semantics as `grounding eval`.
3. `config.example.yaml` gains a top-level `retrieval` section with a `rerank` block exposing `enabled`, `model`, `pool_size`, `batch_size`. Defaults match `RerankConfig` defaults. Inline comments document each field.
4. Config resolution order (documented in `CLAUDE.md`):
   1. Explicit CLI flag value.
   2. `config.yaml` (`retrieval.rerank.*`) when present.
   3. `RerankConfig` default.
5. MCP tool schema for `search_corpus` gains per-field `description` strings for `rerank_enabled`, `rerank_pool_size`, `rerank_model` (plumbed in 18.2 with minimal descriptions). Descriptions explain:
   - `rerank_enabled`: whether to apply cross-encoder reranking after FAISS.
   - `rerank_pool_size`: FAISS candidate count fed to the reranker; higher pool = better quality, slower.
   - `rerank_model`: override the default reranker model name.
6. `CLAUDE.md` gains a "Reranking" sub-section under an appropriate heading (e.g., near "Embedding Generation") covering: when to enable, the latency table from Epic 18's architecture notes, config resolution order, and CLI flag reference.
7. `docs/eval/README.md` gains a short note about the new flags (how to run rerank-on vs rerank-off evals locally). Actual measured comparison is deferred to Story 18.4.
8. Backward compatibility: invocations without any rerank-related flags or config produce bit-for-bit identical behavior to pre-18.3 (opt-in preserved).
9. Validation: flag values go through `RerankConfig.validate()` (18.1) at CLI parse time; invalid values exit with an actionable error and a non-zero code (reuse existing CLI error exit code, e.g., 2 for argument errors).
10. Unit tests extend `tests/test_eval_cli.py` and add `tests/test_local_rag_cli_flags.py` as needed to cover flag parsing, config.yaml resolution, and invalid-value rejection.
11. Epic 16 eval gate stays green (rerank is still off by default; 18.4 is where the flip happens).

### Flag naming consistency

Use identical flag names across `grounding eval` and `local_rag.py`:

| Flag | Type | Default | Behavior |
|------|------|---------|----------|
| `--rerank` | store_true | false | Enables reranking. |
| `--rerank-model` | str | `BAAI/bge-reranker-base` | Model name. |
| `--rerank-pool-size` | int | 50 | FAISS candidate pool before rerank. |
| `--rerank-top-k` | int | None | Override final top-k post-rerank; falls back to `--top-k`. |

The `--rerank-top-k` override exists because some workflows want a larger FAISS pool than post-rerank result count â€” e.g., `--rerank-pool-size 100 --rerank-top-k 5` asks the reranker to score 100 candidates and return the best 5. When `--rerank-top-k` is omitted, the existing `--top-k` applies to the final output. Document this in `--help`.

### Why a shared `grounding/config.py` helper (Task 4)

Two CLIs now read `config.yaml`. Three, if you count the MCP server (though it doesn't need rerank config from yaml â€” it gets it from tool arguments). A tiny helper prevents the two CLIs from diverging on path resolution or key handling. Keep the helper small: no Pydantic, no schemas â€” just `yaml.safe_load` and a couple of `.get(...)` calls.
