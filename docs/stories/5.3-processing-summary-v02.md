## Story Description

As a **developer**, I want to **produce a comprehensive processing summary** so that users see completed, failed, skipped counts, chunk totals, and timing information at the end of every run.

---

## Context

### From PRD v0.2  
[Source: docs/prd.md — Section "Batch Processing" & "Observability"]

- After each run, the CLI must show a clear summary with counts and durations.  
- Failed files should be listed with stage-specific reasons.  
- Chunk totals and elapsed time provide quick verification.

### From Architecture v0.2  
[Source: docs/architecture.md — Section 4 Pipeline]

- `ProcessingStats` already tracks metrics; summary string requires enhancement.  
- Provide consistent formatting that aligns with earlier stories (CLI assertions, QA checks).

---

## Acceptance Criteria

### AC1: Summary Content
- ✅ Summary includes: total files, processed, succeeded, failed, skipped, chunk total, duration, parse/format times.  
- ✅ Failed files listed with stage-prefixed reasons (e.g., `parser: ...`).  
- ✅ Summary remains deterministic for assertions/log review.

### AC2: Chunk Count Tracking
- ✅ Pipeline/controller track `chunk_count` on each `FileContext`.  
- ✅ `ProcessingStats` aggregates real chunk totals (no placeholder values).  
- ✅ Manifest entries store actual chunk counts.

### AC3: CLI Output
- ✅ CLI prints summary after run regardless of success.  
- ✅ Dry-run describes planned actions, summary shows zero chunk totals.  
- ✅ CLI exits with code 1 when failures present (summary reflects counts).

---

## Implementation Highlights

- Added `chunk_count` to `FileContext` and enhanced `ProcessingStats` (`add_chunks`, `record_postprocess_failure`).  
- Controller now handles chunk/metadata/writer/manifest errors per-stage, updates stats, and refreshes progress callbacks.  
- Manifest and stats totals reflect actual chunk counts; failure reasons include stage prefixes.  
- CLI summary practices unchanged but now relies on enriched stats.
