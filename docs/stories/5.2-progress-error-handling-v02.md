## Story Description

As a **developer**, I want to **provide progress visibility and resilient error handling** so that users see the current processing status and the batch continues even when individual files fail.

---

## Context

### From PRD v0.2  
[Source: docs/prd.md — Sections "User Experience" & "Reliability"]

- CLI must expose meaningful progress feedback for long runs.  
- Failures should surface clearly but not stop the entire batch.  
- Summary must list failed files with reasons.

### From Architecture v0.2  
[Source: docs/architecture.md — Sections 4 Pipeline & 8 Error Handling]

- Use tqdm progress bar with file name updates.  
- Wrap per-file processing in try/except to capture stage-specific errors.  
- Update `ProcessingStats` with failure reasons.

### Dependencies
- Depends on controller integration (Story 5.1).  
- Stats/manifest/writer already implemented.

---

## Acceptance Criteria

### AC1: Progress Reporting
- ✅ Display tqdm progress bar with total files, updating per processed PDF.  
- ✅ Progress bar shows current file slug and status (SUCCESS/FAILED).  
- ✅ Dry-run mode logs planned actions without updating the progress bar.

### AC2: Error Isolation
- ✅ Each stage (scanner, parser, formatter, chunker, metadata, writer) wrapped in try/except so a failure marks the file as failed without aborting.  
- ✅ Errors logged with file name, stage, and message.  
- ✅ Remaining PDFs continue processing.

### AC3: Stats & Summary
- ✅ `ProcessingStats` records failure reasons accessible in summary.  
- ✅ CLI summary lists failure counts/messages; exit code non-zero when failures occurred.  
- ✅ Manifests omitted for failed files.

---

## Implementation Highlights

- Extended `run_pipeline` to catch and log per-stage errors while continuing processing.  
- Controller respects `progress_callback` for tqdm updates and skips failed contexts when generating chunk/metadata outputs.  
- CLI progress bar now reflects SUCCESS/FAILED status; dry-run logs planned actions.  
- Stats summary reflects failure counts; CLI preserves exit behavior (non-zero on failure).
