# Story 19.3: Wire Hybrid into Retrieval Paths

**Epic:** Epic 19 - Hybrid Retrieval (BM25 + Dense)
**Status:** Done
**Owner:** Product Owner (Sarah)
**Created:** 2026-04-15
**Dependencies:** Story 19.1 (BM25 index builder), Story 19.2 (hybrid + RRF)
**Related:** ROADMAP Tier 1 #1; parallels Story 18.3 (rerank plumbing)

---

## Story

**As a** grounding-ai user,
**I want** consistent `--hybrid` / `--hybrid-pool-size` / `--hybrid-k-rrf` flags on every CLI, a documented `retrieval.hybrid` block in `config.example.yaml`, a richer MCP tool schema, and a `SearchCorpusTool` / eval-runner integration that calls `search_hybrid` before rerank,
**so that** I can turn on BM25+dense fusion from `grounding eval`, `local_rag.py`, or any MCP client without editing code — and Story 19.4 can measure the `{hybrid × rerank}` matrix through the same code paths users will hit.

---

## Acceptance Criteria

1. `grounding eval` subcommand exposes three hybrid flags: `--hybrid` (bool, default false), `--hybrid-pool-size` (int, default 50), `--hybrid-k-rrf` (int, default 60). Flags use the same naming and semantics across every CLI surface.
2. `scripts/local_rag.py` exposes the same three flags with the same names, defaults, and semantics as `grounding eval`.
3. `config.example.yaml` gains a `retrieval.hybrid` block with `enabled`, `pool_size`, `k_rrf` keys. Defaults match `HybridConfig` defaults (`enabled=false`, `pool_size=50`, `k_rrf=60`). Inline comments document each field and reference `CLAUDE.md`.
4. Config resolution order (documented in `CLAUDE.md`, mirroring Story 18.3 AC4):
   1. Explicit CLI flag value.
   2. `config.yaml` → `retrieval.hybrid.*` when present.
   3. `HybridConfig` default.
5. `grounding/config.py` gains `resolve_hybrid_config(*, retrieval_config, cli_enabled, cli_pool_size, cli_k_rrf)` analogous to `resolve_rerank_config`. It returns a validated `HybridConfig` and raises `ValueError` on invalid values (caller converts to exit code 2, per 18.3 pattern).
6. `SearchCorpusTool` accepts `hybrid_config: HybridConfig | None = None`. When `hybrid_config.enabled` is true, the tool calls `grounding.hybrid.search_hybrid(...)` to produce the candidate pool, enriches each result dict with body / source / page / section via the existing `_read_chunk` helper, and preserves the existing result-dict schema (`rank`, `source`, `content`, `score`, `chunk_id`, `doc_id`, `page_start`, `page_end`, `section_heading`). Additive hybrid-only keys (`faiss_rank`, `bm25_rank`, `rrf_score`, optional `hybrid_degraded`) pass through untouched so downstream consumers can inspect them.
7. MCP `search_corpus` tool schema gains three optional inputs — `hybrid_enabled` (bool, default false), `hybrid_pool_size` (int, default 50, minimum 1), `hybrid_k_rrf` (int, default 60, minimum 1) — with per-field `description` strings in the style of 18.3 AC5. Descriptions explain:
   - `hybrid_enabled`: whether to fuse FAISS + BM25 via RRF before (optional) rerank.
   - `hybrid_pool_size`: candidates fetched from each channel before fusion; higher = better recall, slightly slower.
   - `hybrid_k_rrf`: RRF damping constant (default 60; literature standard). Raising it flattens rank contributions.
8. `grounding/eval/runner.py:run_eval` accepts a new `hybrid_config: HybridConfig | None = None` kwarg. When enabled, the runner calls `search_hybrid` (via an injectable `hybrid_fn` for tests) instead of `_default_search`, then routes the pool through the existing rerank path when `rerank_config.enabled` is also true. An `EvalRun.hybrid` provenance block (mirroring `RerankProvenance`) is emitted when hybrid ran; `None` otherwise.
9. Hybrid + rerank composition: when both are enabled, hybrid produces the candidate pool (sized to `max(hybrid.pool_size, rerank.pool_size, top_k)`) and the reranker re-scores that pool before final truncation. The reranker never sees a dense-only pool when hybrid is on. The citation prefix formats the final top-k in both surfaces.
10. Backward compatibility: invocations without any `--hybrid*` flag and without a `retrieval.hybrid` config block produce bit-for-bit identical behavior to pre-19.3. Verified by the mini-corpus eval gate.
11. Validation: CLI flag values pass through `HybridConfig.validate()` at parse time; invalid values exit with code 2 and an actionable stderr message, reusing the `EXIT_BAD_ARGS = 2` alias 18.3 introduced in `grounding/eval/cli.py`.
12. `CLAUDE.md` gains a "Hybrid Retrieval" sub-section (placed immediately below the existing "Reranking" section) covering: when to enable, interaction with rerank (hybrid runs first), the three CLI flags with the same column shape as the rerank table, the three-tier config resolution order, and a "latency note" row pointing users to 19.4's measured numbers.
13. `docs/eval/README.md` gains a short "Hybrid retrieval (opt-in, Story 19.3)" sub-section with rerank-on-vs-off-style example commands for the `{hybrid off/on} × {rerank off/on}` matrix. Actual measured comparison numbers are deferred to Story 19.4.
14. Unit tests cover: flag parsing and defaults on both CLIs; config.yaml resolution (precedence, missing block, malformed); invalid-value rejection (exit 2); `SearchCorpusTool` hybrid path produces same top-k shape as the dense path; `search_hybrid` integration in `run_eval`; rerank + hybrid composition (rerank receives the hybrid pool, not the dense pool). No real FAISS, BM25, or reranker loads — all injected.
15. Epic 16 CI gate stays green on the mini corpus with no hybrid flag (default off). CI is re-measured in 19.4 with hybrid on.

### Config block (for `config.example.yaml`)

```yaml
  hybrid:
    # Enable BM25 + dense fusion via Reciprocal Rank Fusion before
    # (optional) rerank. Off by default; see CLAUDE.md "Hybrid Retrieval"
    # for when to enable and the latency/quality trade-off.
    enabled: false

    # Candidates fetched from each channel (FAISS and BM25) before fusion.
    # Higher = better recall, slightly slower. Default 50.
    pool_size: 50

    # Reciprocal Rank Fusion damping constant. Literature standard is 60;
    # larger values flatten rank contributions, smaller values amplify the
    # top ranks. Default 60.
    k_rrf: 60
```

### Composition rule (hybrid × rerank)

The flow when both are on:

```
query
  └─► search_hybrid(top_k=pool_k, pool_size=hybrid.pool_size, k_rrf=hybrid.k_rrf)
        └─► RRF-merged pool of up to 2 × hybrid.pool_size unique chunks
              └─► enrich each with body / front matter
                    └─► rerank(query, enriched_pool, config=rerank_cfg)
                          └─► truncate to top_k
                                └─► citation prefix formats the final list
```

`pool_k = max(hybrid.pool_size, rerank.pool_size if rerank_on else 0, top_k)` — this is an explicit design choice: the reranker wants the widest possible pool to reorder, and the hybrid fetch is already the expensive step, so paying once for the max is cheaper than re-fetching. Same reasoning that 18.3's `fetch_k = max(rerank.pool_size, top_k)` uses.

When hybrid is on and rerank is off: `pool_k = max(hybrid.pool_size, top_k)`; result is truncated to `top_k` after RRF.

### `SearchCorpusTool` injection story

The tool currently owns `self.index`, `self.chunk_map`, `self.embedder`, `self.corpus_dir`. For `search_hybrid`, we need an `embeddings_dir` (index location) and the three injectable loaders (`load_index_fn`, `load_bm25_fn`, `embed_fn`) — but `search_hybrid` defaults all three to the real production callables, so the straightforward wiring is: pass `embeddings_dir` through the constructor, and let `search_hybrid` use its defaults. This matches the MCP server's existing pattern of constructing `SearchCorpusTool` with all four fields.

A small wrinkle: `search_hybrid`'s `load_index_fn` default loads a fresh FAISS index from disk, but `SearchCorpusTool` already holds a pre-loaded `self.index` / `self.chunk_map`. To avoid double-loading, pass `load_index_fn=lambda _dir: (self.index, self.chunk_map)` in the hybrid call. The BM25 loader uses its default (loads from `embeddings_dir`). The embedder is the tricky one: `search_hybrid`'s `embed_fn` default uses `grounding.embedder.generate_embedding`, but the tool has `self.embedder` (a `SentenceTransformer` instance). Wrap it: `embed_fn=lambda q: self.embedder.encode([q], normalize_embeddings=True)[0]`. Document the wrapping shape in the Task 4 implementation notes.

### `hybrid_degraded` key-presence convention

Carry-over from 19.2 QA (advisory item on Ready-for-Done): the `hybrid_degraded: True` key is **only present** on result dicts when the BM25 artifacts were missing and `search_hybrid` fell back to dense-only. Downstream code must use `result.get("hybrid_degraded", False)` rather than assuming the key exists. CLAUDE.md's Hybrid Retrieval section codifies this convention so 19.4 and future consumers don't have to re-derive it.

### Why MCP doesn't read `config.yaml` (for hybrid either)

The MCP server already skips `config.yaml` for rerank — it takes rerank kwargs directly from tool arguments because the server runs as a long-lived process with no CLI and no per-call config file. Hybrid follows the same pattern: `hybrid_enabled` / `hybrid_pool_size` / `hybrid_k_rrf` come from the tool invocation, not from disk. This is a deliberate scope fence, not an oversight.

### Backward compatibility invariant

When no `--hybrid*` flag is passed and no `retrieval.hybrid` block exists in `config.yaml`, `_resolve_hybrid_config_from_args` returns `None`, and `SearchCorpusTool.execute` / `run_eval` skip the hybrid branch entirely. The dense-only code path is the same code as before — no re-routing through hybrid with `enabled=False`. Verified via:

- `test_eval_without_hybrid_flag_hybrid_config_is_none`
- `test_local_rag_no_hybrid_flags_no_config_returns_none`
- `test_run_eval_hybrid_config_none_calls_default_search`
- `test_hybrid_config_none_uses_dense_path`
- Mini-corpus eval gate (Task 11): `worst_drop 0.000` against `docs/eval/baselines/mini-corpus.json`.

### Duplication debt (carried forward from 18.3)

Both CLIs now have three near-duplicate resolver helpers (`_resolve_rerank_config_from_args`, `_resolve_hybrid_config_from_args`). 18.3's QA review flagged this as a low-priority maintainability nit and explicitly deferred it. 19.3 inherits the debt rather than paying it off: the right refactor is a single `grounding/config.py` helper that takes the parsed argparse namespace and returns `(rerank_config, hybrid_config)`, with an optional `on_error` callback that both CLIs can wire to their preferred exit mechanism (return code vs `sys.exit`). Tracked informally; not blocking for 19.3. Story 19.4 or a standalone follow-up can unify the three helpers.

### Out of scope

- Measuring the lift and flipping the `retrieval.hybrid.enabled` default → Story 19.4.
- Environment-variable overrides (`GROUNDING_HYBRID_ENABLED=1`) → defer; CLI + config is enough.
- Per-agent hybrid overrides in agent YAMLs → future enhancement.
- HyDE / query rewriting → Tier 2 #5; separate epic.
- Learned fusion weights (α·dense + (1-α)·bm25) → Epic 19 explicitly picks RRF; revisit only if 19.4 measurement demands it.
- Swapping RRF for a different fusion algorithm → out of scope.
- Snippet highlighting or search-UI affordances → out of scope.

Story 19.3 does not measure hybrid latency — by design, deferred to Story 19.4. The off-by-default flip and the latency note in CLAUDE.md cross-linking the rerank latency table mitigate the risk for users who don't opt in. CONCERNS, not FAIL: the open question is "what is the cost," not "is the cost prohibitive."
